В этом документе описываются основные шаги по настройке Odoo на производстве или на сервере, подключенном к Интернету. Следует installation, и, как правило, не требуется для систем разработки, не представленных в Интернете.

Предупреждение

Если вы настраиваете публичный сервер, обязательно ознакомьтесь с нашими рекомендациями Безопасность!

dbfilter

Odoo - это мультитенантная система: одна система Odoo может работать и обслуживать несколько экземпляров базы данных. Он также очень настраиваемый, с настройками (начиная с загружаемых модулей) в зависимости от «текущей базы данных».

Это не проблема при работе с бэкэндом (веб-клиентом) в качестве зарегистрированного пользователя компании: базу данных можно выбрать при входе в систему, а затем загрузить настройки.

Однако это проблема для незарегистрированных пользователей (портал, веб-сайт), которые не связаны с базой данных: Odoo необходимо знать, какую базу данных следует использовать для загрузки страницы веб-сайта или выполнения операции. Если многопользовательский режим не используется, это не проблема, можно использовать только одну базу данных, но если доступно несколько баз данных, Odoo необходимо правило, чтобы знать, какую из них использовать.

Это одна из целей: option: –db-filter <odoo-bin –db-filter> `: указывает, как следует выбирать базу данных на основе запрашиваемого имени хоста (домена). Значением является `регулярное выражение, возможно, включающее динамически внедренное имя хоста (` % h`) или первый поддомен (` % d`), через который осуществляется доступ к системе.

Для серверов, на которых размещено несколько баз данных, особенно если используется `` website``, dbfilter ** должен быть установлен **, иначе ряд функций будет работать некорректно.

Образцы конфигурации

  • Показывать только базы данных с именами, начинающимися с «mycompany»

в `` / etc / odoo.conf`` установлено:

[options]
dbfilter = ^mycompany.*$
  • Отображать только базы данных, соответствующие первому поддомену после `` www``: например, база данных «mycompany» будет отображаться, если входящий запрос был отправлен на «www.mycompany.com» или «mycompany.co.uk» , но не для `` www2.mycompany.com`` или `` helpdesk.mycompany.com``.

в `` / etc / odoo.conf`` установлено:

[options]
dbfilter = ^%d$

Примечание

Установка правильного: option: --db-filter <odoo-bin --db-filter> `является важной частью обеспечения вашего развертывания. Если он работает правильно и соответствует только одной базе данных для каждого имени хоста, настоятельно рекомендуется заблокировать доступ к экранам менеджера баз данных и использовать параметр запуска `--no-database-list` для предотвращения вывода списка ваших баз данных, и заблокировать доступ к экранам управления базой данных. Смотрите также security.

PostgreSQL

По умолчанию PostgreSQL разрешает соединение только через сокеты UNIX и петлевые соединения (от «localhost», на той же машине, на которой установлен сервер PostgreSQL).

Гнездо UNIX хорошо, если вы хотите, чтобы Odoo и PostgreSQL выполнялись на одной и той же машине, и используется по умолчанию, когда хост не указан, но если вы хотите, чтобы Odoo и PostgreSQL выполнялись на разных машинах [# различных машин] _, это необходимо прослушивать сетевые интерфейсы [# remote-socket] _, либо:

  • Принимайте только петлевые соединения и используйте SSH туннель между машиной, на которой работает Odoo, и той, на которой работает PostgreSQL, затем настройте Odoo для подключения к своему концу туннеля.
  • Примите подключения к машине, на которой установлен Odoo, возможно, через ssl (подробнее см. «Настройки подключения PostgreSQL»), затем настройте Odoo для подключения по сети.

Пример конфигурации

  • Разрешить TCP-соединение на локальном хосте
  • Разрешить TCP-соединение из сети 192.168.1.x

в `` / etc / postgresql / 9.5 / main / pg_hba.conf`` установлено:

# IPv4 local connections:
host    all             all             127.0.0.1/32            md5
host    all             all             192.168.1.0/24          md5

в `` / etc / postgresql / 9.5 / main / postgresql.conf`` установлено:

listen_addresses = 'localhost,192.168.1.2'
port = 5432
max_connections = 80

Конфигурирование Odoo

Из коробки Odoo подключается к локальному postgres через сокет UNIX через порт 5432. Это можно изменить, используя :ref:`the database options <reference/cmdline/server/database>`когда ваше развертывание Postgres не является локальным и / или не использует установки по умолчанию.

: Ref: упакованные установщики <setup/install/packaged> `автоматически создаст нового пользователя ( odoo`) и установит его в качестве пользователя базы данных.

  • Экраны управления базой данных защищены настройкой `` admin_passwd``. Этот параметр может быть установлен только с помощью файлов конфигурации, и его просто проверяют перед выполнением изменений базы данных. Следует установить случайное значение, чтобы третьи лица не могли использовать этот интерфейс.
  • Все операции с базой данных используют database options, включая экран управления базой данных. Для работы экрана управления базой данных требуется, чтобы пользователь PostgreSQL имел право `` createb``.

  • Пользователи всегда могут удалить свои базы данных. Чтобы экран управления базой данных полностью не функционировал, пользователь PostgreSQL должен быть создан с помощью `` no-creatb``, а база данных должна принадлежать другому пользователю PostgreSQL.

    Предупреждение

    пользователь PostgreSQL * не должен * быть суперпользователем

Пример конфигурации

  • подключиться к серверу PostgreSQL на 192.168.1.2
  • порт 5432
  • используя учетную запись пользователя «odoo»,
  • с паролем в качестве пароля
  • фильтрация только БД с именем, начинающимся с 'mycompany'

в `` / etc / odoo.conf`` установлено:

[options]
admin_passwd = mysupersecretpassword
db_host = 192.168.1.2
db_port = 5432
db_user = odoo
db_password = pwd
dbfilter = ^mycompany.*$

Встроенный сервер

Odoo включает в себя встроенные HTTP-серверы, использующие многопоточность или многопроцессорность.

Для производственного использования рекомендуется использовать многопроцессорный сервер, поскольку он повышает стабильность, несколько лучше использует вычислительные ресурсы и может лучше контролироваться и ограничиваться ресурсами.

  • Многопроцессорность включается путем настройки: option: `ненулевое количество рабочих процессов <odoo-bin –workers> `, число рабочих должно быть основано на количестве ядер в машине (возможно, с некоторым количеством рабочих для cron в зависимости от того, сколько прогнозируется работа cron)
  • Рабочие ограничения могут быть настроены на основе конфигурации оборудования, чтобы избежать исчерпания ресурсов

Предупреждение

многопроцессорный режим в настоящее время недоступен в Windows

Расчет рабочего числа

  • Правило большого пальца: (#CPU * 2) + 1
  • Cron работникам нужен процессор
  • 1 работник ~ = 6 одновременно работающих пользователей

расчет объема памяти

  • Мы считаем, что 20% запросов - это тяжелые запросы, а 80% - более простые.
  • A heavy worker, when all computed field are well designed, SQL requests are well designed, … is estimated to consume around 1GB of RAM
  • Легкий рабочий, по тому же сценарию, по оценкам, потребляет около 150 МБ ОЗУ

Требуется ОЗУ = # работник * ((light_worker_ratio * light_worker_ram_estima) + (heavy_worker_ratio * heavy_worker_ram_esvaluation))

Живой чат

В многопроцессорном режиме выделенный рабочий LiveChat автоматически запускается и прослушивает: option: `longpolling port <odoo-bin –longpolling-port> `но клиент не будет подключаться к нему.

Вместо этого у вас должен быть прокси-сервер, перенаправляющий запросы, URL-адрес которых начинается с `` / longpolling / `` на порт longpolling. Другой запрос должен быть перенаправлен на: option: `normal HTTP port <odoo-bin –http-port> `

Чтобы добиться этого, вам необходимо развернуть обратный прокси-сервер перед Odoo, например, nginx или apache. При этом вам нужно будет перенаправить еще несколько http-заголовков в Odoo и активировать proxy_mode в конфигурации Odoo, чтобы Odoo прочитал эти заголовки.

Пример конфигурации

  • Сервер с 4 процессорами, 8 потоков
  • 60 одновременных пользователей
  • 60 пользователей / 6 = 10 &lt;- теоретически необходимое количество работников
  • (4 * 2) + 1 = 9 &lt;- теоретическое максимальное количество работников
  • Мы будем использовать 8 рабочих + 1 для cron. Мы также будем использовать систему мониторинга для измерения нагрузки на процессор и проверки, находится ли она между 7 и 7,5.
  • RAM = 9 * ((0,8 * 150) + (0,2 * 1024)) ~ = 3Go RAM для Odoo

в `` / etc / odoo.conf``:

[options]
limit_memory_hard = 1677721600
limit_memory_soft = 629145600
limit_request = 8192
limit_time_cpu = 600
limit_time_real = 1200
max_cron_threads = 1
workers = 8

HTTPS

Доступ к нему осуществляется через веб-сайт / веб-клиент или веб-сервис, Odoo передает информацию об аутентификации в открытом виде. Это означает, что для безопасного развертывания Odoo необходимо использовать HTTPS [#switching] _. Завершение SSL может быть реализовано через любой прокси-сервер завершения SSL, но требует следующей настройки:

  • Включить Odoo's: опция: `режим прокси <odoo-bin –proxy-mode> `. Это должно быть включено только когда Odoo находится за обратным прокси
  • Настройте прокси-сервер завершения SSL (Пример завершения Nginx)
  • Настройте сам прокси (пример прокси Nginx)
  • Ваш прокси-сервер завершения SSL также должен автоматически перенаправлять незащищенные соединения на безопасный порт

Пример конфигурации

  • Перенаправить http запросы на https
  • Прокси-запросы к odoo

в `` / etc / odoo.conf`` установлено:

proxy_mode = True

в `` / etc / nginx / sites-enabled / odoo.conf`` установлено:

#odoo server
upstream odoo {
 server 127.0.0.1:8069;
}
upstream odoochat {
 server 127.0.0.1:8072;
}

# http -> https
server {
   listen 80;
   server_name odoo.mycompany.com;
   rewrite ^(.*) https://$host$1 permanent;
}

server {
 listen 443;
 server_name odoo.mycompany.com;
 proxy_read_timeout 720s;
 proxy_connect_timeout 720s;
 proxy_send_timeout 720s;

 # Add Headers for odoo proxy mode
 proxy_set_header X-Forwarded-Host $host;
 proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
 proxy_set_header X-Forwarded-Proto $scheme;
 proxy_set_header X-Real-IP $remote_addr;

 # SSL parameters
 ssl on;
 ssl_certificate /etc/ssl/nginx/server.crt;
 ssl_certificate_key /etc/ssl/nginx/server.key;
 ssl_session_timeout 30m;
 ssl_protocols TLSv1 TLSv1.1 TLSv1.2;
 ssl_ciphers 'ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-AES256-GCM-SHA384:DHE-RSA-AES128-GCM-SHA256:DHE-DSS-AES128-GCM-SHA256:kEDH+AESGCM:ECDHE-RSA-AES128-SHA256:ECDHE-ECDSA-AES128-SHA256:ECDHE-RSA-AES128-SHA:ECDHE-ECDSA-AES128-SHA:ECDHE-RSA-AES256-SHA384:ECDHE-ECDSA-AES256-SHA384:ECDHE-RSA-AES256-SHA:ECDHE-ECDSA-AES256-SHA:DHE-RSA-AES128-SHA256:DHE-RSA-AES128-SHA:DHE-DSS-AES128-SHA256:DHE-RSA-AES256-SHA256:DHE-DSS-AES256-SHA:DHE-RSA-AES256-SHA:AES128-GCM-SHA256:AES256-GCM-SHA384:AES128-SHA256:AES256-SHA256:AES128-SHA:AES256-SHA:AES:CAMELLIA:DES-CBC3-SHA:!aNULL:!eNULL:!EXPORT:!DES:!RC4:!MD5:!PSK:!aECDH:!EDH-DSS-DES-CBC3-SHA:!EDH-RSA-DES-CBC3-SHA:!KRB5-DES-CBC3-SHA';

Odoo как приложение WSGI

Также возможно монтировать Odoo как стандартное приложение WSGI. Odoo предоставляет базу для сценария запуска WSGI как `` odoo-wsgi.example.py``. Этот сценарий должен быть настроен (возможно, после копирования его из каталога установки) для правильной настройки конфигурации непосредственно в: mod: odoo.tools.config, а не через командную строку или файл конфигурации.

Однако сервер WSGI будет предоставлять только главную конечную точку HTTP для веб-клиента, веб-сайта и API веб-сервиса. Поскольку Odoo больше не контролирует создание рабочих, он не может настроить cron или livechat работников.

Cron Workers

Для запуска заданий cron для развертывания Odoo в качестве приложения WSGI требуется

  • Классический Odoo (запускается через `` odoo-bin``)
  • Подключен к базе данных, в которой должны выполняться задания cron (через: option: odoo-bin -d)
  • Который не должен подвергаться воздействию сети. Чтобы гарантировать, что бегуны cron недоступны по сети, можно полностью отключить встроенный HTTP-сервер с помощью: option: odoo-bin --no-http или установки` http_enable = False` в файле конфигурации

Живой чат

Второй проблемной подсистемой для развертываний WSGI является LiveChat: там, где большинство HTTP-соединений относительно короткие и быстро освобождают свой рабочий процесс для следующего запроса, LiveChat требует долгоживущего соединения для каждого клиента для реализации уведомлений почти в реальном времени. ,

Это противоречит рабочей модели на основе процессов, поскольку она связывает рабочие процессы и препятствует доступу новых пользователей к системе. Однако эти долгоживущие соединения делают очень мало и в основном остаются на стоянке в ожидании уведомлений.

Решения для поддержки livechat / мотивации в приложении WSGI:

  • Разверните многопоточную версию Odoo (экземпляр первичной обработки на основе процесса) и перенаправьте только запросы к URL-адресам, начинающимся с `` / longpolling / ``, на этот Odoo, это самый простой, а URL-адрес с длинным опросом может удвоиться как экземпляр cron ,
  • Разверните чётный Odoo через `` odoo-gevent`` и прокси-запросы, начинающиеся с `` / longpolling / `` to: option: `longpolling port <odoo-bin –longpolling-port> `.

Обслуживание статических файлов

Для удобства разработки Odoo напрямую обслуживает все статические файлы в своих модулях. Это может быть не идеально, когда речь идет о производительности, и статические файлы обычно должны обслуживаться статическим HTTP-сервером.

Статические файлы Odoo находятся в папке `` static / `` каждого модуля, поэтому статические файлы могут быть обработаны путем перехвата всех запросов: samp: / {MODULE} / static / {FILE} и поиска нужного модуля (и файл) в различных путях аддонов.

Безопасность

Для начала имейте в виду, что защита информационной системы - это непрерывный процесс, а не разовая операция. В любой момент вы будете в безопасности только в качестве самого слабого звена в своей среде.

Поэтому, пожалуйста, не принимайте этот раздел как окончательный список мер, которые предотвратят все проблемы с безопасностью. Это всего лишь краткое изложение первых важных вещей, которые вы должны обязательно включить в свой план действий по обеспечению безопасности. Все остальное будет связано с лучшими практиками безопасности для вашей операционной системы и дистрибутива, лучшими практиками в отношении пользователей, паролей, управления доступом и т. Д.

При развертывании сервера, подключенного к Интернету, обязательно учитывайте следующие темы, связанные с безопасностью:

  • Всегда устанавливайте надежный пароль администратора супер-администратора и ограничивайте доступ к страницам управления базой данных, как только система настроена. Смотрите Безопасность менеджера баз данных.
  • Выбирайте уникальные логины и надежные пароли для всех учетных записей администраторов во всех базах данных. Не используйте «admin» в качестве логина. Не используйте эти логины для повседневных операций, только для контроля / управления установкой. Никогда не используйте пароли по умолчанию, такие как admin / admin, даже для тестовых / промежуточных баз данных.
  • ** Не ** устанавливайте демонстрационные данные на интернет-серверах. Базы данных с демонстрационными данными содержат логины и пароли по умолчанию, которые можно использовать для входа в ваши системы и вызвать серьезные проблемы даже в системах staging / dev.
  • Используйте соответствующие фильтры базы данных (: опция: --db-filter <odoo-bin --db-filter> `) ограничить видимость ваших баз данных в соответствии с именем хоста. Смотрите :ref:`db_filter. Вы также можете использовать: option: `-d <odoo-bin -d> `предоставить собственный (разделенный запятыми) список доступных баз данных для фильтрации, вместо того, чтобы позволить системе извлекать их все из базы данных базы данных.
  • После того, как ваши `` db_name`` и `` db_filter`` настроены и соответствуют только одной базе данных для каждого имени хоста, вы должны установить опцию конфигурации `` list_db`` в `` False``, чтобы полностью запретить перечисление баз данных и заблокировать доступ к экранам управления базой данных (это также отображается как опция: `–no-database-list <odoo-bin –no-database-list> `опция командной строки)
  • Убедитесь, что пользователь PostgreSQL (: option: --db_user <odoo-bin --db_user> `) * не * суперпользователь, и ваши базы данных принадлежат другому пользователю. Например, они могут принадлежать суперпользователю ` postgres``, если вы используете выделенный непривилегированный `` db_user``. Смотрите также Конфигурирование Odoo.
  • Обновляйте установки, регулярно устанавливая последние сборки, либо через GitHub, либо загружая последнюю версию с https://www.odoo.com/page/download или http://nightly.odoo.com.
  • Настройте сервер в многопроцессорном режиме с надлежащими ограничениями, соответствующими типичному использованию (память / ЦП / тайм-ауты). Смотрите также Встроенный сервер.
  • Запустите Odoo за веб-сервером, обеспечивающим завершение HTTPS действительным сертификатом SSL, чтобы предотвратить подслушивание при передаче открытого текста. SSL-сертификаты дешевы, и существует множество бесплатных опций. Настройте веб-прокси, чтобы ограничить размер запросов, установите соответствующие таймауты, а затем включите параметр: option: proxy mode <odoo-bin --proxy-mode> `вариант. Смотрите также :ref:`https_proxy.
  • Если вам нужно разрешить удаленный SSH-доступ к вашим серверам, обязательно установите надежный пароль для ** всех ** учетных записей, а не только для пользователя root. Настоятельно рекомендуется полностью отключить аутентификацию на основе пароля и разрешить только аутентификацию с открытым ключом. Также подумайте об ограничении доступа через VPN, разрешении только доверенных IP-адресов в брандмауэре и / или использовании системы обнаружения перебором, такой как fail2ban или эквивалентной.

  • Рассмотрите возможность установки соответствующего ограничения скорости на вашем прокси или брандмауэре, чтобы предотвратить атаки методом грубой силы и атаки типа отказ в обслуживании. Смотрите также Блокировка атак грубой силы для конкретных мер.

    Многие сетевые провайдеры предоставляют автоматическую защиту от атак распределенного отказа в обслуживании (DDOS), но это часто дополнительная услуга, поэтому вам следует проконсультироваться с ними.

  • По возможности размещайте общедоступные демонстрационные / тестовые / промежуточные экземпляры на компьютерах, отличных от производственных. И применять те же меры безопасности, что и для производства.
  • Если вы размещаете несколько клиентов, изолируйте данные и файлы клиентов друг от друга, используя контейнеры или соответствующие методы «тюрьмы».
  • Настраивайте ежедневные резервные копии ваших баз данных и данных хранилища файлов и копируйте их на удаленный сервер архивации, который недоступен с самого сервера.

Блокировка атак грубой силы

Для развертываний с выходом в Интернет очень распространены атаки методом грубой силы на пароли пользователей, и этой угрозой нельзя пренебрегать для серверов Odoo. Odoo создает запись в журнале всякий раз, когда выполняется попытка входа в систему, и сообщает результат: успех или неудача вместе с целевым именем входа и исходным IP-адресом.

Записи в журнале будут иметь следующую форму.

Не удалось войти

2018-07-05 14:56:31,506 24849 INFO db_name odoo.addons.base.res.res_users: Login failed for db:db_name login:admin from 127.0.0.1

Успешный вход в систему:

2018-07-05 14:56:31,506 24849 INFO db_name odoo.addons.base.res.res_users: Login successful for db:db_name login:admin from 127.0.0.1

Эти журналы могут быть легко проанализированы системой предотвращения вторжений, такой как fail2ban.

Например, следующее определение фильтра fail2ban должно соответствовать неудачной регистрации:

[Definition]
failregex = ^ \d+ INFO \S+ \S+ Login failed for db:\S+ login:\S+ from <HOST>
ignoreregex =

Это может быть использовано с определением тюрьмы для блокировки атакующего IP по HTTP (S).

Вот как это может выглядеть для блокировки IP-адреса на 15 минут, когда с одного и того же IP-адреса в течение 1 минуты обнаруживаются 10 неудачных попыток входа в систему:

[odoo-login]
enabled = true
port = http,https
bantime = 900  ; 15 min ban
maxretry = 10  ; if 10 attempts
findtime = 60  ; within 1 min  /!\ Should be adjusted with the TZ offset
logpath = /var/log/odoo.log  ;  set the actual odoo log path here

Безопасность менеджера баз данных

Конфигурирование Odoo mentioned admin_passwd in passing.

Этот параметр используется на всех экранах управления базами данных (для создания, удаления, сброса или восстановления баз данных).

Если экраны управления вообще не должны быть доступны, вы должны установить для параметра конфигурации `` list_db`` значение `` False``, чтобы заблокировать доступ ко всем экранам выбора и управления базой данных.

Предупреждение

Настоятельно рекомендуется отключить менеджер баз данных для любой интернет-системы! Он предназначен как инструмент разработки / демонстрации, позволяющий легко создавать базы данных и управлять ими. Он не предназначен для использования в производстве и может даже представлять опасные функции для злоумышленников. Он также не предназначен для работы с большими базами данных и может вызывать ограничения памяти.

В производственных системах операции по управлению базами данных всегда должны выполняться системным администратором, включая предоставление новых баз данных и автоматическое резервное копирование.

Обязательно установите соответствующий параметр `` db_name`` (и, возможно, тоже `` db_filter``), чтобы система могла определять целевую базу данных для каждого запроса, в противном случае пользователи будут заблокированы, поскольку им не будет позволено выбирать сами базы данных.

Если экраны управления должны быть доступны только с выбранного набора компьютеров, используйте функции прокси-сервера, чтобы заблокировать доступ ко всем маршрутам, начинающимся с `` / web / database``, за исключением (возможно) `` / web / database / selector`` который отображает экран выбора базы данных.

Если экран управления базой данных следует оставить доступным, параметр `` admin_passwd`` должен быть изменен с его значения по умолчанию `` admin``: этот пароль проверяется перед разрешением операций по изменению базы данных.

Он должен храниться надежно и генерироваться случайным образом, например

$ python3 -c 'import base64, os; print(base64.b64encode(os.urandom(24)))'

который сгенерирует псевдослучайную печатную строку из 32 символов.

Поддерживаемые браузеры

Odoo поддерживается несколькими браузерами для каждой из своих версий. Никаких различий не делается в соответствии с версией браузера, чтобы быть в курсе. Odoo поддерживается в текущей версии браузера. Мы поддерживаем следующие браузеры:

  • Гугл Хром
  • Сафари
