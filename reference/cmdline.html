Запуск сервера

-d <database>, --database <database>

базы данных, используемые при установке или обновлении модулей. Предоставление списка через запятую ограничивает доступ к базам данных, указанным в списке.

Для расширенных параметров базы данных, посмотрите below.

-i <modules>, --init <modules>

Разделенный запятыми список модулей, которые нужно установить перед запуском сервера (требуется: option: -d).

-u <modules>, --update <modules>

Разделенный запятыми список модулей для обновления перед запуском сервера (требуется: option: -d).

--addons-path <directories>

разделенный запятыми список каталогов, в которых хранятся модули. Эти каталоги сканируются на наличие модулей.

-c <config>, --config <config>

предоставить альтернативный configuration file

-s, --save

сохраняет конфигурацию сервера в текущий файл конфигурации (по умолчанию:: file: {$ HOME} /. odoorc и может быть переопределен с помощью: option:` -c`).

--without-demo

отключает загрузку демонстрационных данных для модулей, установленных через запятую, используйте `` all`` для всех модулей.

--test-enable

запускает тесты после установки модулей

--test-tags 'tag_1,tag_2,...,-tag_n'

выберите тесты для запуска с помощью тегов.

--screenshots

Укажите каталог, в который нужно записывать снимки экрана при сбое теста HttpCase.browser_js. По умолчанию это: file: / tmp / odoo_tests / {db_name} / screenshots

--screencasts

Включите скринкасты и укажите каталог для записи файлов скринкастов. Для кодирования кадров в видеофайл необходимо установить утилиту `` ffmpeg``. В противном случае кадры будут сохранены вместо видеофайла.

`` 1``, `` t`` или `` true`` могут использоваться для использования того же каталога, что и вышеупомянутая опция --screenshots`.

База данных

-r <user>, --db_user <user>

имя пользователя базы данных, используется для подключения к PostgreSQL.

-w <password>, --db_password <password>

пароль базы данных, если используется аутентификация по паролю.

--db_host <hostname>

хост для сервера базы данных

  • `` localhost`` в Windows
  • Сокет UNIX в противном случае
--db_port <port>

порт, который прослушивает база данных, по умолчанию 5432

--db-filter <filter>

скрывает базы данных, которые не соответствуют `` <filter> ``. Фильтр является регулярным выражением, с дополнениями, которые:

  • % h заменяется на полное имя хоста, на которое сделан запрос.

  • % d заменяется поддоменом, на который делается запрос, за исключением `` www`` (поэтому домен `` odoo.com`` и `` www.odoo.com`` оба соответствуют базе данных `` odoo``).

    Эти операции чувствительны к регистру. Добавьте параметр `` (? I) `` для сопоставления всех баз данных (чтобы домен `` odoo.com`` с использованием `` (? I)% d`` совпадал с базой данных `` Odoo``).

Начиная с версии 11, также можно ограничить доступ к данному прослушиванию базы данных, используя параметр –database и указав список баз данных через запятую

При объединении двух параметров db-фильтр заменяет список баз данных через запятую для ограничения списка баз данных, а список через запятую используется для выполнения запрошенных операций, таких как обновление модулей.

$ odoo-bin --db-filter ^11.*$

Ограничить доступ к базам данных, имя которых начинается с 11

$ odoo-bin --database 11firstdatabase,11seconddatabase

Ограничить доступ только к двум базам данных: 11firstdatabase и 11seconddatabase

$ odoo-bin --database 11firstdatabase,11seconddatabase -u base

Ограничьте доступ только к двум базам данных, 11firstdatabase и 11seconddatabase, и обновите базовый модуль для одной базы данных: 11firstdatabase. Если база данных 11seconddatabase не существует, база данных создается и базовые модули устанавливаются

$ odoo-bin --db-filter ^11.*$ --database 11firstdatabase,11seconddatabase -u base

Ограничьте доступ к базам данных, имя которых начинается с 11, и обновите базовый модуль для одной базы данных: 11firstdatabase. Если база данных 11seconddatabase не существует, база данных создается и базовые модули устанавливаются

--db-template <template>

при создании новых баз данных из экранов управления базами данных используйте указанный шаблон базы данных. По умолчанию используется `` template0``.

--pg_path </path/to/postgresql/binaries>

Путь к двоичным файлам PostgreSQL, которые используются менеджером баз данных для создания и восстановления баз данных. Вы должны указать эту опцию, только если эти двоичные файлы находятся в нестандартном каталоге.

--no-database-list

Подавляет возможность перечисления баз данных, доступных в системе

--db_sslmode

Управляйте безопасностью SSL соединения между Odoo и PostgreSQL. Значение должно быть равно «отключить», «разрешить», «предпочесть», «потребовать», «проверить-ca» или «проверить-заполнить». Значение по умолчанию - «предпочесть»

Сообщения электронной почты

--email-from <address>

Адрес электронной почты используется как <FROM> когда Odoo нужно отправлять письма

--smtp <server>

Адрес SMTP-сервера для подключения для отправки писем

--smtp-port <port>
--smtp-ssl

Если установлено, odoo должен использовать SMTP-соединения SSL / STARTSSL

--smtp-user <name>

Имя пользователя для подключения к SMTP-серверу

--smtp-password <password>

Пароль для подключения к SMTP-серверу

интернационализация

Используйте эти опции для перевода Odoo на другой язык. Смотрите раздел i18n в руководстве пользователя. Опция -d обязательна. Опция '-l' обязательна в случае импорта

--load-language <languages>

указывает языки (разделенные запятыми) для переводов, которые вы хотите загрузить

-l, --language <language>

указать язык файла перевода. Используйте его с –i18n-export или –i18n-import

--i18n-export <filename>

экспортировать все предложения для перевода в файл CSV, PO-файл или архив TGZ и выйти.

--i18n-import <filename>

импортировать файл CSV или PO с переводами и выйти. Опция -l обязательна.

--i18n-overwrite

перезаписывает существующие термины перевода при обновлении модуля или импорте файла CSV или PO.

--modules

указать модули для экспорта. Использовать в сочетании с –i18n-export

Расширенные настройки

Особенности разработчика

--dev <feature,feature,...,feature>
  • `` all``: все функции ниже активированы
  • `` xml``: читать шаблон qweb из файла xml напрямую, а не из базы данных. Как только шаблон был изменен в базе данных, он не будет считываться из файла xml до следующего обновления / init.
  • `` reload``: перезапустить сервер после обновления файла python (может не обнаруживаться в зависимости от используемого текстового редактора)
  • `` qweb``: перерыв в оценке шаблона qweb, когда узел содержит `` t-debug = 'debugger'``
  • `` (i) p (u) db``: запустить выбранный отладчик python в коде, когда возникла непредвиденная ошибка перед регистрацией и возвратом ошибки.

HTTP

--no-http

не запускайте HTTP или работников с длинным опросом (все равно можете запустить: ref: `cron <reference/actions/cron> работники)

Предупреждение

не действует, если задано: option: --test-enable, так как тесты требуют доступного HTTP-сервера

--http-interface <interface>

TCP / IP-адрес, на котором слушает HTTP-сервер, по умолчанию `` 0.0.0.0`` (все адреса)

--http-port <port>

Порт, по которому слушает HTTP-сервер, по умолчанию - 8069.

--longpolling-port <port>

Порт TCP для соединений с длительным опросом в многопроцессорном режиме или режиме Gevent, по умолчанию - 8072. Не используется в режиме по умолчанию (многопоточном).

--proxy-mode

позволяет использовать заголовки `` X-Forwarded- * `` через поддержку прокси Werkzeug.

Предупреждение

режим прокси * не должен * быть включен вне сценария обратного прокси

логирование

По умолчанию Odoo отображает все журналы уровня `` info``, кроме ведения журнала рабочего процесса (только `` warning``), и вывод журнала отправляется в `` stdout``. Доступны различные параметры для перенаправления ведения журнала в другие места назначения и для настройки объема вывода журнала.

--logfile <file>

отправляет вывод журнала в указанный файл вместо стандартного вывода. В Unix файл может управляться внешними программами ротации журналов и будет автоматически открыт при замене

--syslog

регистрирует в системном журнале событий: syslog on unices и` Журнал событий в Windows <https://docs.python.org/3/library/logging.handlers.html#nteventloghandler>`_.

Ни один не настраивается

--log-db <dbname>

регистрирует модель `` ir.logging`` (таблица `` ir_logging``) указанной базы данных. База данных может быть именем базы данных в «текущем» PostgreSQL или URI PostgreSQL, например, для агрегации журналов.

--log-handler <handler-spec>

: samp: {LOGGER}: {LEVEL}, включает `` LOGGER`` в указанном `` LEVEL``, например, `` odoo.models: DEBUG`` включит все сообщения регистрации в или выше `` DEBUG`` уровень в моделях.

  • Двоеточие ``: `` обязательно
  • Регистратор может быть опущен для настройки корневого (по умолчанию) обработчика
  • Если уровень опущен, регистратор устанавливается на `` INFO``

Опцию можно повторить для настройки нескольких регистраторов, например

$ odoo-bin --log-handler :DEBUG --log-handler werkzeug:CRITICAL --log-handler odoo.fields:WARNING
--log-request

включить ведение журнала DEBUG для запросов RPC, эквивалентно `` –log-handler = odoo.http.rpc.request: DEBUG``

--log-response

включить ведение журнала DEBUG для ответов RPC, эквивалентно `` –log-handler = odoo.http.rpc.response: DEBUG``

--log-web

включает ведение журнала DEBUG HTTP-запросов и ответов, что эквивалентно `` –log-handler = odoo.http: DEBUG``

--log-sql

включает ведение журнала DEBUG для SQL-запросов, что эквивалентно `` –log-handler = odoo.sql_db: DEBUG``

--log-level <level>

Ярлык для более легкой установки предопределенных уровней на определенных регистраторах. «реальные» уровни (`` критический``, `` ошибка``, `` warn``, `` debug``) установлены на регистраторах `` odoo`` и `` werkzeug`` (кроме `` debug``, который устанавливается только на `` odoo``).

Odoo также предоставляет отладочные псевдоуровни, которые применяются к различным наборам регистраторов:

`` Debug_sql``

устанавливает регистратор SQL в `` debug``

эквивалентно `` –log-sql``

`` Debug_rpc``

устанавливает для `` odoo`` и регистраторов HTTP-запросов значение `` debug``

эквивалентно `` –log-level debug –log-request``

`` Debug_rpc_answer``

устанавливает для `` odoo`` и HTTP-регистраторов запросов и ответов значение `` debug``

эквивалентно `` –log-level debug –log-request –log-response``

Примечание

В случае конфликта между: option: --log-level и: option:` –log-handler`, используется последний

многопроцессорная обработка

--workers <count>

если значение `` count`` не равно 0 (по умолчанию), включает многопроцессорность и устанавливает указанное количество рабочих HTTP (подпроцессы, обрабатывающие запросы HTTP и RPC).

Примечание

многопроцессорный режим доступен только в Unix-системах

Ряд опций позволяет ограничивать и перерабатывать работников:

--limit-request <limit>

Количество запросов, которые рабочий обработает перед тем, как их переработать и перезапустить.

По умолчанию * 8196 *.

--limit-memory-soft <limit>

Максимально допустимая виртуальная память на одного работника. Если лимит превышен, работник уничтожается и перерабатывается в конце текущего запроса.

По умолчанию * 2048MiB *.

--limit-memory-hard <limit>

Жесткий лимит на виртуальную память, любой работник, превышающий лимит, будет немедленно уничтожен, не дожидаясь окончания текущей обработки запроса.

По умолчанию * 2560MiB *.

--limit-time-cpu <limit>

Предотвращает использование работником более <limit> Секунды процессора для каждого запроса. Если предел превышен, работник убит.

По умолчанию * 60 *.

--limit-time-real <limit>

Предотвращает работника от принятия дольше, чем <limit> секунд, чтобы обработать запрос. Если предел превышен, работник убит.

Отличается от: option: --limit-time-cpu тем, что это ограничение` времени стены `, включая, например, запросы SQL.

По умолчанию * 120 *.

--max-cron-threads <count>

количество работников, посвященных: ref: `cron <reference/actions/cron> рабочие места. По умолчанию * 2 *. Рабочие являются потоками в многопоточном режиме и обрабатывают в многопроцессорном режиме.

Для режима мультиобработки это в дополнение к рабочим процессам HTTP.

Файл конфигурации

Большинство параметров командной строки также можно указать через файл конфигурации. В большинстве случаев они используют похожие имена с удаленным префиксом `` -, а другие `` - заменяются на `` _`, например: option: `--db-template становится` db_template` ,

Некоторые преобразования не соответствуют шаблону:

  • : option: --db-filter становится` dbfilter`
  • : option: --no-http соответствует логическому` http_enable`
  • предустановки ведения журнала (все опции, начинающиеся с `` –log-, за исключением: option: `--log-handler` и: option:` --log-db`) просто добавляют содержимое в `` log_handler, используйте что прямо в файле конфигурации
  • : option: --smtp хранится как` smtp_server`
  • : option: --database хранится как` db_name`
  • : option: --i18n-import и: option:` –i18n-export` вообще не доступны из файлов конфигурации

Файл конфигурации по умолчанию: file: {$ HOME} /. Odoorc, который можно переопределить, используя: option:` –config <odoo-bin -c> `. Указание: option: `–save <odoo-bin -s> `сохранит текущее состояние конфигурации обратно в этот файл.

Ракушка

Командная строка Odoo также позволяет запускать odoo как консольную среду Python. Это позволяет напрямую взаимодействовать с: ref: `orm <reference/orm> и его функциональные возможности.

$ odoo_bin shell
--shell-interface (ipython|ptpython|bpython|python)

Укажите предпочитаемый REPL для использования в режиме оболочки.

подмости

Scaffolding - это автоматическое создание каркасной структуры для упрощения начальной загрузки (новых модулей, в случае Odoo). Хотя в этом нет необходимости, это позволяет избежать утомительной настройки базовых структур и выяснения всех начальных требований.

Строительные леса доступны через подкоманду: command: odoo-bin scaffold.

name (required)

имя создаваемого модуля может быть изменено различными способами для генерации программных имен (например, имя каталога модуля, названия модели,…)

destination (default=current directory)

каталог, в котором создается новый модуль, по умолчанию используется текущий каталог

-t <template>

каталог шаблонов, файлы передаются через jinja2, затем копируются в каталог `` destination``

$ odoo_bin scaffold my_module /addons/

Это создаст модуль * my_module * в каталоге * / addons / *.
