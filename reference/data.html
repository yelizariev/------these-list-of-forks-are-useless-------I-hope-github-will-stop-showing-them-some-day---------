Odoo в значительной степени управляется данными, и большая часть определения модулей, таким образом, является определением различных записей, которыми он управляет: пользовательский интерфейс (меню и представления), безопасность (права доступа и правила доступа), отчеты и простые данные - все определяются через записи ,

Структура

Основным способом определения данных в Odoo является использование файлов данных XML. Общая структура файла данных XML заключается в следующем:

  • Любое количество элементов операции в корневом элементе `` odoo``
<!-- the root elements of the data file -->
<?xml version="1.0" encoding="UTF-8"?>
<odoo>
  <operation/>
  ...
</odoo>

Файлы данных выполняются последовательно, операции могут ссылаться только на результат операций, определенных ранее.

Примечание

Если предполагается, что содержимое файла данных будет применено только один раз, вы можете указать флаг odoo `` noupdate``, установленный на 1. Если часть данных в файле предполагается применить один раз, вы можете поместить эту часть файла в <data noupdate=»1»> домен.

<odoo>
    <data noupdate="1">
        <!-- Only loaded when installing the module (odoo-bin -i module) -->
        <operation/>
    </data>

    <!-- (Re)Loaded at install and update (odoo-bin -i/-u) -->
    <operation/>
</odoo>

Основные операции

`` Record``

`` record`` соответствующим образом определяет или обновляет запись в базе данных, она имеет следующие атрибуты:

`` модель`` (обязательно)
название модели для создания (или обновления)
`` Id``

: term: external identifier для этой записи. Настоятельно рекомендуется предоставить один

  • для создания записи, позволяет последующим определениям либо изменять, либо ссылаться на эту запись
  • для изменения записи, запись для изменения
`` context``
контекст для использования при создании записи
`` forcecreate``

в режиме обновления, должна ли быть создана запись, если она не существует

Требует: term: external id, по умолчанию` True`.

`` field``

Каждая запись может состоять из тегов `` field``, определяющих значения, которые нужно установить при создании записи. `` Record`` без `` field`` будет использовать все значения по умолчанию (создание) или ничего не делать (обновление).

`` Field`` имеет обязательный атрибут `` name``, имя устанавливаемого поля и различные методы для определения самого значения:

Ничего
если для поля не указано значение, в поле будет установлен неявный `` False``. Может использоваться для очистки поля или для избежания использования значения по умолчанию для поля.
`` Search``

для relational fields, должен быть доменом :ref:`relational fields <reference/fields/relational>`на полевой модели.

Оценит домен, найдет модель поля, используя ее, и установит результат поиска в качестве значения поля. Первый результат будет использоваться только в том случае, если поле является Many2one

`` ref``

если указан атрибут `` ref``, его значение должно быть действительным: term: external id, который будет найден и установлен в качестве значения поля.

В основном для полей Many2one и Many2one

`` type``

если указан атрибут `` type``, он используется для интерпретации и преобразования содержимого поля. Содержимое поля может быть предоставлено через внешний файл с помощью атрибута `` file`` или через тело узла.

Доступные типы:

`` xml``, `` html``
извлекает потомков `` field`` как отдельный документ, оценивает любой: term: external id, указанный в форме` % (external_id) s`. `` %% `` может использоваться для вывода фактических *% * знаков.
`` File``
гарантирует, что содержимое поля является допустимым путем к файлу в текущей модели, сохраняет пару: samp: {module}, {path} в качестве значения поля
`` char``
устанавливает содержимое поля непосредственно как значение поля без изменений
`` base64``
base64-кодирует содержимое поля, полезно в сочетании с атрибутом `` file`` * * для загрузки, например, данных изображения во вложения
`` int``
преобразует содержимое поля в целое число и устанавливает его в качестве значения поля
`` Float``
преобразует содержимое поля в число с плавающей точкой и устанавливает его в качестве значения поля
`` список``, `` tuple``
должно содержать любое количество элементов `` value`` с теми же свойствами, что и у `` field``, каждый элемент разрешается в элемент сгенерированного кортежа или списка, а сгенерированная коллекция устанавливается как значение поля
`` eval``

для случаев, когда предыдущие методы не подходят, атрибуты `` eval`` просто оценивают любое предоставленное им выражение Python и устанавливают результат в качестве значения поля.

Контекст оценки содержит различные модули (`` time``, `` datetime``, `` timedelta``, lativedelta), функцию для разрешения: term:` external identifiers` (`` ref``) и объект модели для текущего поля, если применимо (`` obj``)

`` Delete``

Тег `` delete`` может удалить любое количество записей, определенных ранее. Он имеет следующие атрибуты:

`` модель`` (обязательно)
модель, в которой указанная запись должна быть удалена
`` Id``
: term: external id удаляемой записи
`` Search``
a :ref:`domain <reference/orm/domains>`найти записи модели для удаления

`` id`` и `` search`` являются эксклюзивными

`` function``

Тег `` function`` вызывает метод для модели с предоставленными параметрами. Он имеет два обязательных параметра `` model`` и `` name``, определяющих соответственно модель и имя вызываемого метода.

Параметры могут быть предоставлены с использованием `` eval`` (должна оцениваться последовательность параметров для вызова метода) или элементов `` value`` (см. Значения `` list``).

<odoo>
    <data noupdate="1">
        <record name="partner_1" model="res.partner">
            <field name="name">Odude</field>
        </record>

        <function model="res.partner" name="send_inscription_notice"
            eval="[[ref('partner_1'), ref('partner_2')]]"/>

        <function model="res.users" name="send_vip_inscription_notice">
            <function eval="[[('vip','=',True)]]" model="res.partner" name="search"/>
        </function>
    </data>

    <record id="model_form_view" model="ir.ui.view">

    </record>
</odoo>

Ярлыки

Поскольку некоторые важные структурные модели Odoo являются сложными и вовлеченными, файлы данных предоставляют более короткие альтернативы их определению с использованием record tags:

`` menuitem``

Определяет запись `` ir.ui.menu`` с рядом значений по умолчанию и откатов:

`` parent``
  • Если установлен атрибут `` parent``, это должен быть: term: external id другого элемента меню, используемого в качестве родителя нового элемента
  • Если `` parent`` не предоставлен, он пытается интерпретировать атрибут `` name`` как `` / `` -разделенную последовательность имен меню и находит место в иерархии меню. В этой интерпретации промежуточные меню создаются автоматически
  • В противном случае меню определяется как элемент меню «верхнего уровня» (* не * меню без родительского элемента)
`` Name``
Если атрибут `` name`` не указан, пытается получить имя меню из связанного действия, если оно есть. В противном случае используется запись `` id``
`` groups``
Атрибут `` groups`` интерпретируется как разделенная запятыми последовательность: term: external identifiers для моделей` res.groups`. Если префикс: term: external identifier стоит с минусом (` -`), группа * удаляется * из групп меню
`` Action``
если указан, атрибут `` action`` должен быть: term: external id действия, которое нужно выполнить при открытом меню
`` Id``
пункт меню: термин: внешний идентификатор

`` template``

Создает QWeb view <reference/views/qweb>`требуется только arch` раздел представления и разрешено несколько * необязательных * атрибутов:

`` Id``
представление: термин: внешний идентификатор
`` name``, `` inherit_id``, `` priority``
аналогично соответствующему полю в `` ir.ui.view`` (примечание: `` attributeit_id`` должен быть: term: external identifier)
`` primary``
если установлено в `` True`` и в сочетании с `` inherit_id``, определяет представление как первичное
`` groups``
список группы через запятую: термин: внешние идентификаторы
`` Page``
если установлено значение `` True ``, шаблон является страницей веб-сайта (ссылка на которую можно удалить)
`` Optional``
`` enabled`` или `` disabled``, может ли представление быть отключено (в интерфейсе сайта) и его статус по умолчанию. Если не установлено, представление всегда включено.

`` report``

Создает :ref:`IrActionsReport <reference/actions/report>`запись с несколькими значениями по умолчанию.

Чаще всего просто передает атрибуты в соответствующие поля в `` ir.actions.report``, но также автоматически создает элемент в меню: guilabel: More` модели отчета.

Файлы данных XML являются гибкими и информативными, но очень многословными при массовом создании нескольких простых записей одной и той же модели.
