В этом разделе объясняется, как предоставить переводческие возможности вашему модулю.

Примечание

Если вы хотите внести свой вклад в перевод самого Odoo, пожалуйста, обратитесь к странице Odoo Wiki.

Экспортируемый переводимый термин

В результате ряд терминов в ваших модулях «неявно переводим», даже если вы не выполнили какую-либо конкретную работу по переводу, вы можете экспортировать переводимые термины вашего модуля и найти контент для работы.

Экспорт переводов выполняется через интерфейс администрирования, войдя в интерфейс бэкэнда и открыв: menuselection: Настройки -&gt; Переводы -&gt; Импорт / Экспорт -&gt; Экспорт переводов

  • оставить язык по умолчанию (новый язык / пустой шаблон)
  • выберите формат PO File
  • выберите свой модуль
  • нажмите: guilabel: Export и загрузите файл

Это дает вам файл с именем: file: {yourmodule} .pot, который следует переместить в каталог: file:` {yourmodule} / i18n / . Файл представляет собой * PO Template *, в котором просто перечисляются переводимые строки и из которого можно создавать фактические переводы (PO-файлы). PO-файлы могут быть созданы с помощью msginit_, с помощью специального инструмента перевода, такого как POEdit_, или просто скопировав шаблон в новый файл с именем: file: `{language} .po. Файлы перевода должны быть помещены в: file: {yourmodule} / i18n /, рядом с: file: {yourmodule} .pot, и будут автоматически загружаться Odoo при установке соответствующего языка (через Settings ‣ Translations ‣ Languages)

Примечание

переводы для всех загруженных языков также устанавливаются или обновляются при установке или обновлении модуля

Неявный экспорт

Odoo автоматически экспортирует переводимые строки из содержимого типа «данные»:

  • в представлениях, отличных от QWeb, экспортируются все текстовые узлы, а также содержимое атрибутов `` string``, `` help``, `` sum``, `` verify`` и `` placeholder``
  • Шаблоны QWeb (как на стороне сервера, так и на стороне клиента), экспортируются все текстовые узлы, кроме внутри блоков `` t-translation = off , содержимого `` title, `` alt``, ` Атрибуты label` и` placeholder` также экспортируются

  • for Field, если их модель не помечена как` _translate = False`:

    • их атрибуты `` string`` и `` help`` экспортируются
    • если присутствует `` selection`` и список (или кортеж), он экспортируется
    • если их атрибут `` translate`` установлен на `` True``, все их существующие значения (по всем записям) экспортируются
  • Справка / сообщения об ошибках: attr: ~ odoo.models.Model._constraints и: attr:` ~ odoo.models.Model._sql_constraints` экспортируются

Явный экспорт

Когда дело доходит до более «императивных» ситуаций в коде Python или коде Javascript, Odoo не может автоматически экспортировать переводимые термины, поэтому они должны быть явно помечены для экспорта. Это делается путем переноса литеральной строки в вызов функции.

В Python функция обтекания выглядит так odoo._()

title = _("Bank Accounts")

В JavaScript функция обертки обычно выглядит так: js odoo.web._t():

title = _t("Bank Accounts");

Предупреждение

Для экспорта могут быть помечены только литеральные строки, а не выражения или переменные. В ситуациях, когда строки форматируются, это означает, что строка формата должна быть помечена, а не форматированная строка

Ленивая версия _ и` _t` - это odoo._lt() в python и: js odoo._lt() в javascript. Поиск перевода выполняется только при рендеринге и может использоваться для объявления переводимых свойств в методах класса глобальных переменных.

переменные

** Не ** экстракт может работать, но он не будет правильно переводить текст

_("Scheduled meeting with %s" % invitee.name)

** Do ** установить динамические переменные вне поиска перевода

_("Scheduled meeting with %s") % invitee.name

Блоки

** Не ** разделяйте ваш перевод на несколько блоков или несколько строк

# bad, trailing spaces, blocks out of context
_("You have ") + len(invoices) + _(" invoices waiting")
_t("You have ") + invoices.length + _t(" invoices waiting");

# bad, multiple small translations
_("Reference of the document that generated ") + \
_("this sales order request.")

** Do ** держите в одном блоке, предоставляя полный контекст для переводчиков

# good, allow to change position of the number in the translation
_("You have %s invoices wainting") % len(invoices)
_.str.sprintf(_t("You have %s invoices wainting"), invoices.length);

# good, full sentence is understandable
_("Reference of the document that generated " + \
  "this sales order request.")

множественное число

** Не ** множить термины по-английски

msg = _("You have %s invoice") % invoice_count
if invoice_count > 1:
  msg += _("s")

** Имейте в виду, что каждый язык имеет различные формы множественного числа:

if invoice_count > 1:
  msg = _("You have %s invoices") % invoice_count
else:
  msg = _("You have %s invoice") % invoice_count

Читать против времени выполнения

** Не ** вызывать поиск перевода при запуске сервера

ERROR_MESSAGE = {
  # bad, evaluated at server launch with no user language
  'access_error': _('Access Error'),
  'missing_error': _('Missing Record'),
}

class Record(models.Model):

  def _raise_error(self, code):
    raise UserError(ERROR_MESSAGE[code])

** Не ** вызывать перевод поиска, когда файл JavaScript читается

# bad, js _t is evaluated too early
var core = require('web.core');
var _t = core._t;
var map_title = {
    access_error: _t('Access Error'),
    missing_error: _t('Missing Record'),
};

** Используйте ** метод поиска ленивых переводов

ERROR_MESSAGE = {
  'access_error': _lt('Access Error'),
  'missing_error': _lt('Missing Record'),
}

class Record(models.Model):

  def _raise_error(self, code):
    # translation lookup executed at error rendering
    raise UserError(ERROR_MESSAGE[code])

или ** делать ** динамически оценивать переводимый контент

# good, evaluated at run time
def _get_error_message(self):
  return {
    access_error: _('Access Error'),
    missing_error: _('Missing Record'),
  }

** Выполните ** в случае, когда поиск перевода выполняется, когда файл JS * read *, используйте _lt вместо` _t`, чтобы перевести термин, когда он * используется *

# good, js _lt is evaluated lazily
var core = require('web.core');
var _lt = core._lt;
var map_title = {
    access_error: _lt('Access Error'),
    missing_error: _lt('Missing Record'),
};
