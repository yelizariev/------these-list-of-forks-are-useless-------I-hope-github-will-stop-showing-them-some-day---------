Views are what define how records should be displayed to end-users. They are specified in XML which means that they can be edited independently from the models that they represent. They are flexible and allow a high level of customization of the screens that they control. There exist various types of views. Each of them represents a mode of visualization: form, list, kanban, etc.

Generic structure

Basic views generally share the common structure defined below. Placeholders are denoted in all caps.

<record id="MODEL_view_TYPE" model="ir.ui.view">
  <field name="name">NAME</field>
  <field name="model">MODEL</field>
  <field name="arch" type="xml">
    <VIEW_TYPE>
      <VIEW_SPECIFICATIONS/>
    </VIEW_TYPE>
  </field>
</record>

Fields

View objects expose a number of fields. They are optional unless specified otherwise.

  • name (mandatory) Char

    Only useful as a mnemonic/description of the view when looking for one in a list of some sort.

  • model Char

    The model linked to the view, if applicable.

  • priority Integer

    When a view is requested by (model, type), the view matching the model and the type, with the lowest priority will be returned (it is the default view).

    It also defines the order of views application during view inheritance.

  • groups_id Many2many -> odoo.addons.base.models.res_users.Groups

    The groups allowed to use/access the current view.

    If the view extends an existing view, the extension will only be applied for a given user if the user has access to the provided groups_id.

  • arch Text

    The description of the view layout.

Attributes

The different view types have a wide variety of attributes allowing customizations of the generic behaviors. Some main attributes will be explained here. They do not all have an impact on all view types.

Примечание

The current context and user access rights may also impact the view abilities.

  • `` Create``

    Disable/enable record creation on the view.

  • edit (form & list & gantt)

    Disable/enable record edition on the view.

  • delete (form & list)

    Disable/enable record deletion on the view through the Action dropdown.

  • duplicate (form & list)

    Disable/enable record duplication on the view through the Action dropdown.

  • decoration-$ (list & gantt)

    Define a conditional display of a record in the style of a row’s text based on the corresponding record’s attributes.

    Values are Python expressions. For each record, the expression is evaluated with the record’s attributes as context values and, if true, the corresponding style is applied to the row. Other context values are uid (the id of the current user) and current_date (the current date as a string of the form YYYY-MM-DD).

  decoration-danger="state == 'help_needed'"
  decoration-bf="state='busy'">
  <TREE_VIEW_CONTENT>
</tree>

    Предупреждение

    Supported values differ for the two view types. The Gantt view only supports success, info, warning, danger and secondary displays. The list view supports bf, it, success, info, warning, danger, muted and primary displays.

наследование

Inheritance fields

The two following View fields are used to specify inherited views.

  • inherit_id Many2one

    the current view’s parent view, unset by default

  • mode Selection: extension / primary

    inheritance mode, extension: by default if inherit_id is set, primary otherwise.

Просмотр соответствия

  • if a view is requested by (model, type), the view with the right model and type, mode=primary and the lowest priority is matched.
  • when a view is requested by id, if its mode is not primary its closest parent with mode primary is matched.

Разрешение экрана

Разрешение генерирует окончательный `` arch`` для запрашиваемого / совпавшего `` primary`` представления:

  1. если у представления есть родитель, родитель полностью разрешен, тогда применяются спецификации наследования текущего представления
  2. если представление не имеет родителя, его `` arch`` используется как есть
  3. дочерние элементы текущего представления с режимом `` extension`` ищутся, и их спецификации наследования применяются в первую очередь на глубину (применяется дочернее представление, затем его дочерние элементы, затем его родные элементы)

Результат применения дочерних представлений приводит к финальному `` arch``

Характеристики наследования

Спецификации наследования состоят из локатора элемента, который соответствует унаследованному элементу в родительском представлении, и дочернего элемента, который будет использоваться для изменения унаследованного элемента.

Существует три типа локаторов элементов для сопоставления целевого элемента:

  • Элемент `` xpath`` с атрибутом `` expr``. `` expr`` - это выражение XPath [#hasclass] _, примененное к текущему `` arch``, первый найденный узел - это совпадение
  • элемент `` field`` с атрибутом `` name`` сопоставляет первое поле `` field`` с тем же `` name``. Все остальные атрибуты игнорируются при сопоставлении
  • любой другой элемент: первый элемент с тем же именем и идентичными атрибутами (игнорируя атрибуты `` position`` и `` version``)
<xpath expr="page[@name='pg']/group[@name='gp']/field" position="inside">
  <field name="description"/>
</xpath>

<field name="res_id" position="after"/>

<div name="name" position="replace">
  <div name="name2">
    <field name="name2"/>
  </div>
</div>

Спецификация наследования может иметь необязательный атрибут `` position``, определяющий, как должен изменяться соответствующий узел:

`` inside`` (по умолчанию)
содержимое спецификации наследования добавляется к соответствующему узлу
`` replace``
содержимое спецификации наследования заменяет соответствующий узел. Любой текстовый узел, содержащий только `` $ 0`` внутри содержимого спецификации, будет заменен полной копией сопоставленного узла, эффективно оборачивая сопоставляемый узел.
`` after``
содержимое спецификации наследования добавляется к родительскому узлу соответствующего узла после соответствующего узла
`` before``
содержимое спецификации наследования добавляется к родительскому узлу соответствующего узла перед соответствующим узлом
`` attributes``

содержимое спецификации наследования должно быть элементами `` attribute`` с атрибутом `` name`` и необязательным телом:

  • если элемент `` attribute`` имеет тело, новый атрибут, названный по имени его `` name``, создается на соответствующем узле с текстом элемента `` attribute`` в качестве значения
  • если элемент `` attribute`` не имеет тела, атрибут, названный в честь его `` name``, удаляется из соответствующего узла. Если такого атрибута не существует, возникает ошибка
<field name="sale_information" position="attributes">
  <attribute name="invisible">0</attribute>
  <attribute name="attrs">
    {'invisible': [('sale_ok', '=', False)], 'readonly': [('editable', '=', False)]}
  </attribute>
</field>
move

can be used as a direct child of a inheritance spec with a inside, replace, after or before position attribute to move a node.

<xpath expr="//@target" position="after">
    <xpath expr="//@node" position="move"/>
</xpath>

<field name="target_field" position="after">
    <field name="my_field" position="move"/>
</field>

Мероприятия

Представление «Активность» используется для отображения действий, связанных с записями. Данные отображаются на диаграмме с записями, образующими строки, а типы действий - столбцами. В первой ячейке каждой строки отображается (настраиваемая, см. `` Templates``), очень похожая на Kanban) карта, представляющая соответствующую запись. При нажатии на другие ячейки отображается подробное описание всех действий того же типа для записи.

Предупреждение

Представление Activity доступно только в том случае, если установлен модуль `` mail``, и для моделей, которые наследуются от `` mail.activity.mixin``.

Корневым элементом представления Activity является `` <activity> ``, он принимает следующие атрибуты:

  • `` строка`` (обязательно)
    Название, которое должно описывать вид

Возможные дочерние элементы элемента вида:

`` field``

объявляет поля для использования в деятельности * логика *. Если поле просто отображается в виде действия, его не нужно предварительно объявлять.

Возможные атрибуты:

`` имя`` (обязательно)
название поля для выборки
`` templates``

определяет шаблоны Qweb. Определение карт может быть разбито на несколько шаблонов для ясности, но представления активности * должны * определять хотя бы один корневой шаблон `` Activity-Box``, который будет отображаться один раз для каждой записи.

В представлении активности используется в основном стандартный javascript qweb <reference/qweb/javascript>`и предоставляет следующие контекстные переменные (см. ref: reference / views / kanban` для более подробной информации):

`` widget``
current: js ActivityRecord, может использоваться для получения некоторой мета-информации. Эти методы также доступны непосредственно в контексте шаблона и не требуют доступа через `` widget``
`` Record``
объект со всеми запрошенными полями в качестве его атрибутов. Каждое поле имеет два атрибута: `` значение`` и `` raw_value``

Календарь

Представления календаря отображают записи в виде событий в ежедневном, еженедельном или ежемесячном календаре. Их корневой элемент `` <calendar> ``. Доступные атрибуты в представлении календаря:

`` date_start`` (обязательно)
имя поля записи, содержащего дату начала события
`` date_stop``
имя поля записи, содержащего дату окончания события, если указано `` date_stop``, записи становятся подвижными (посредством перетаскивания) непосредственно в календаре
`` date_delay``
альтернатива `` date_stop``, предоставляет продолжительность события вместо даты его окончания (единица: день)
`` color``
имя поля записи, используемого для * цветовой сегментации *. Записи в одном и том же цветном сегменте выделяются одинаковым цветом подсветки в календаре, цвета распределяются полуслучайно. Отображение отображаемого имени / аватара видимой записи на боковой панели
`` form_view_id``
открыть, когда пользователь создает или редактирует событие. Обратите внимание, что если этот атрибут не установлен, представление календаря будет возвращаться к идентификатору представления формы в текущем действии, если оно есть.
`` event_open_popup``
Если для опции 'event_open_popup' установлено значение true, то представление календаря будет открывать события (или записи) в FormViewDialog. В противном случае он откроет события в новом представлении формы (с помощью do_action)
`` Quick_add``
включает быстрое создание события по клику: только запрашивает у пользователя `` имя`` и пытается создать новое событие только с этим и временем события по клику. Откат к полному диалоговому окну, если быстрое создание не удается
`` All_day``
имя логического поля в записи, указывающее, помечено ли соответствующее событие как однодневное (и длительность не имеет значения)
`` mode``
Режим отображения по умолчанию при загрузке календаря. Возможные атрибуты: `` day``, `` week``, `` month``
`` <field> ``

объявляет поля для агрегирования или использования в канбан * логика *. Если поле просто отображается в календаре карты.

Поля могут иметь дополнительные атрибуты:

  • invisible use «True» to hide the value in the cards
  • avatar_field only for x2many field, to display the avatar instead the display_name in the cards
  • write_model and write_field you can add a filter and save the result in the defined model, the filter is added in the sidebar
`` templates``

определяет Qweb шаблон` calendar-box`. Определение карт может быть разбито на несколько шаблонов для ясности, которые будут представлены один раз для каждой записи.

когорта

Enterprise feature

Когортное представление используется для отображения и понимания того, как некоторые данные изменяются в течение определенного периода времени. Например, представьте, что для данного бизнеса клиенты могут подписаться на какой-либо сервис. Затем в когортном представлении может отображаться общее количество подписок в месяц и изучаться частота, с которой клиент покидает службу (отток). Если щелкнуть ячейку, представление когорты перенаправит вас на новое действие, в котором вы увидите только записи, содержащиеся во временном интервале ячейки; это действие содержит представление списка и представление формы.

Примечание

По умолчанию в когортном представлении будут использоваться те же представления списка и формы, что и в действии. Вы можете передать представление списка и представление формы в контекст действия, чтобы установить / переопределить представления, которые будут использоваться (контекстными ключами для использования являются form_view_id и` list_view_id`)

Например, вот очень простое когортное представление:

<cohort string="Subscription" date_start="date_start" date_stop="date" interval="month"/>

Корневым элементом представления Cohort является <cohort> , он принимает следующие атрибуты:

  • `` строка`` (обязательно)
    Название, которое должно описывать вид
  • `` date_start`` (обязательно)
    Действительное поле даты или даты и времени. Это поле понимается представлением как дата начала записи.
  • `` date_stop`` (обязательно)
    Действительное поле даты или даты и времени. Это поле понимается представлением как дата окончания записи. Это поле, которое будет определять отток.
  • `` mode`` (необязательно)
    Строка для описания режима. Это должно быть либо «отток», либо «удержание» (по умолчанию). Режим оттока будет начинаться с 0% и накапливаться со временем, тогда как удержание будет начинаться с 100% и уменьшаться со временем.
  • `` график`` (необязательно)
    Строка для описания временной шкалы. Это должно быть либо «назад», либо «вперед» (по умолчанию). Прямая временная шкала будет отображать данные от date_start до date_stop, тогда как обратная временная шкала будет отображать данные от date_stop до date_start (когда date_start находится в будущем / больше чем date_stop).
  • `` interval`` (необязательно)
    Строка для описания временного интервала. Это должно быть «день», «неделя», «месяц» (по умолчанию) или «год».
  • `` measure`` (необязательно)
    Поле, которое можно агрегировать. Это поле будет использоваться для вычисления значений для каждой ячейки. Если не установлено, в когортном представлении будет подсчитано количество вхождений.

Приборная доска

Enterprise feature

Как и сводное представление и представление графика, представление панели мониторинга используется для отображения агрегированных данных. Тем не менее, на панели инструментов могут быть вложенные представления, что позволяет более полно и интересно взглянуть на данный набор данных.

Представление панели мониторинга может отображать вложенные представления, агрегаты для некоторых полей (по домену) или даже * формулы * (выражения, включающие один или несколько агрегатов). Например, вот очень простая панель инструментов:

<dashboard>
    <view type="graph" ref="sale_report.view_order_product_graph"/>
    <group string="Sale">
        <aggregate name="price_total" field="price_total" widget="monetary"/>
        <aggregate name="order_id" field="order_id" string="Orders"/>
        <formula name="price_average" string="Price Average"
            value="record.price_total / record.order_id" widget="percentage"/>
    </group>
    <view type="pivot" ref="sale_report.view_order_product_pivot"/>
</dashboard>

Корневым элементом представления Dashboard является <dashboard> , он не принимает никаких атрибутов.

Существует 5 возможных типов тегов на панели инструментов:

`` View``

объявляет подпредставление

Допустимые атрибуты:

  • `` type`` (обязательно)
    Тип вложенного представления. Например, * graph * или * pivot *.
  • `` ref`` (необязательно)
    Идентификатор xml для представления. Если не указан, будет использоваться вид по умолчанию для модели.
  • `` name`` (необязательно)
    Строка, которая идентифицирует этот элемент. Это в основном полезно для использования в качестве цели для xpath.
`` group``

определяет макет столбца. На самом деле это очень похоже на элемент группы в представлении формы.

Допустимые атрибуты:

  • `` строка`` (необязательно)
    Описание, которое будет отображаться как заголовок группы.
  • `` colspan`` (необязательно)
    Количество подколонок в этом групповом теге. По умолчанию 6.
  • `` col`` (необязательно)
    Количество столбцов, охватываемых этим групповым тегом (имеет смысл только внутри другой группы). По умолчанию 6.
`` aggregate``

объявляет совокупность. Это значение агрегата для данного поля в текущем домене.

Обратите внимание, что агрегаты должны использоваться внутри группового тега (иначе стиль не будет применен должным образом).

Допустимые атрибуты:

  • `` field`` (обязательно)

    Имя поля, которое нужно использовать для вычисления агрегата. Возможные типы полей:

    • `` integer`` (оператором группы по умолчанию является сумма)
    • `` float`` (оператором группы по умолчанию является сумма)
    • `` many2one`` (оператор группы по умолчанию считается отдельно)
  • `` имя`` (обязательно)
    Строка для идентификации этого агрегата (полезно для формул)
  • `` строка`` (необязательно)
    Краткое описание, которое будет отображаться над значением. Если не указан, он вернется к строке поля.
  • `` домен`` (необязательно)
    Дополнительное ограничение на набор записей, которые мы хотим агрегировать. Этот домен будет объединен с текущим доменом.
  • `` domain_label`` (необязательно)
    Когда пользователь нажимает на агрегат с доменом, он будет добавлен в представление поиска как фасет. Строка, отображаемая для этого фасета, может быть настроена с помощью этого атрибута.
  • `` group_operator`` (необязательно)

    Действительный идентификатор агрегирующей функции postgreSQL, который будет использоваться при агрегировании значений (см. Https://www.postgresql.org/docs/9.5/static/functions-aggregate.html). Если не указано иное, по умолчанию используется group_operator из определения поля. Обратите внимание, что агрегация значений полей не достигается, если значение group_operator равно ``.

    Примечание

    Специальная агрегатная функция `` count_distinct`` (определена в odoo) также может быть использована здесь

    <aggregate name="price_total_max" field="price_total" group_operator="max"/>
  • `` col`` (необязательно)
    Количество столбцов, охватываемых этим тегом (имеет смысл только внутри группы). По умолчанию 1.
  • `` widget`` (необязательно)
    Виджет для форматирования значения (например, атрибут виджета для полей). Например, денежный.
  • `` помощь`` (необязательно)
    Справочное сообщение для всплывающей подсказки во всплывающей подсказке (эквивалент справки для поля в Python)
  • `` measure`` (необязательно)

    Этот атрибут является именем поля, описывающего меру, которая должна использоваться в диаграмме и сводных представлениях при нажатии на агрегат. Специальное значение __count__ может использоваться, чтобы использовать меру счета.

    <aggregate name="total_ojects" string="Total Objects" field="id" group_operator="count" measure="__count__"/>
  • `` clickable`` (необязательно)
    Логическое значение, указывающее, должен ли этот агрегат быть кликабельным или нет (по умолчанию true). Нажатие на кликабельную статистику изменит показатели, используемые подпредставлениями, и добавит значение атрибута домена (если есть) в представление поиска.
  • `` value_label`` (необязательно)
    Строка, расположенная справа от совокупного значения. Например, может быть полезно указать единицу измерения совокупного значения.
`` formula``

объявляет производное значение. Формулы - это значения, рассчитанные из агрегатов.

Обратите внимание, что как и агрегаты, формулы должны использоваться внутри тега группы (иначе стиль не будет применен должным образом).

Допустимые атрибуты:

  • `` значение`` (обязательно)
    Строковое выражение, которое будет оцениваться с помощью встроенного средства оценки Python (в веб-клиенте). Каждый агрегат может использоваться в контексте, в переменной `` record``. Например, `` record.price_total / record.order_id``.
  • `` name`` (необязательно)
    Строка для идентификации этой формулы
  • `` строка`` (необязательно)
    Краткое описание, которое будет отображаться над формулой.
  • `` col`` (необязательно)
    Количество столбцов, охватываемых этим тегом (имеет смысл только внутри группы). По умолчанию 1.
  • `` widget`` (необязательно)
    Виджет для форматирования значения (например, атрибут виджета для полей). Например, денежный. По умолчанию это «float».
  • `` помощь`` (необязательно)
    Справочное сообщение для всплывающей подсказки во всплывающей подсказке (эквивалент справки для поля в Python)
  • `` value_label`` (необязательно)
    Строка, расположенная справа от значения формулы. Например, может быть полезно указать единицу измерения значения формулы.
`` widget``

Объявляет специализированный виджет, который будет использоваться для отображения информации. Этот механизм похож на виджеты в виде формы.

Допустимые атрибуты:

  • `` имя`` (обязательно)
    Строка, определяющая, какой виджет должен быть создан. Представление будет смотреть в `` widget_registry``, чтобы получить правильный класс.
  • `` col`` (необязательно)
    Количество столбцов, охватываемых этим тегом (имеет смысл только внутри группы). По умолчанию 1.

Form

Представления формы используются для отображения данных из одной записи. Их корневой элемент `` <form> ``. Они состоят из обычного HTML с дополнительными структурными и семантическими компонентами.

Структурные компоненты

Структурные компоненты обеспечивают структуру или визуальные функции с небольшой логикой. Они используются в качестве элементов или наборов элементов в представлениях формы.

`` notebook``

определяет раздел с вкладками. Каждая вкладка определяется через дочерний элемент `` page``. Страницы могут иметь следующие атрибуты:

  • string (required) the title of the tab
  • accesskey an HTML accesskey
  • attrs standard dynamic attributes based on record values

Примечание

Обратите внимание, что `` notebook`` не следует размещать внутри `` group``

`` group``

используется для определения макетов столбцов в формах. По умолчанию группы определяют 2 столбца, и большинство прямых дочерних элементов групп занимают один столбец. Прямые дочерние элементы `` field`` по умолчанию отображают метку, а метка и само поле имеют colspan 1.

Количество столбцов в «группе» можно настроить с помощью атрибута «col», количество столбцов, взятых элементом, можно настроить с помощью «colspan».

Дочерние элементы расположены горизонтально (пытается заполнить следующий столбец перед изменением строки).

Группы могут иметь атрибут `` string``, который отображается как заголовок группы

`` Newline``
полезно только в элементах `` group``, досрочно завершает текущую строку и немедленно переключается на новую строку (без предварительного заполнения любого оставшегося столбца)
`` Separator``
небольшой горизонтальный интервал с атрибутом `` string`` ведет себя как заголовок раздела
`` sheet``
может использоваться как прямой дочерний элемент для `` формы`` для более узкой и более отзывчивой формы
`` Header``
в сочетании с `` sheet`` обеспечивает полную ширину над самим листом, обычно используется для отображения кнопок рабочего процесса и виджетов состояния

Enterprise feature

Представления Ганта соответственно отображают диаграммы Ганта (для планирования).

Корневым элементом представлений Ганта является `` <gantt/> ``, он не имеет дочерних элементов, но может принимать следующие атрибуты:

`` date_start`` (обязательно)
имя поля, предоставляющего дату начала и время события для каждой записи.
`` date_stop`` (обязательно)
имя поля, в котором указана конечная продолжительность события для каждой записи.
`` color``
название поля, используемого для окраски таблеток в соответствии с его значением
`` Украшения - {$ имя} ``

разрешить изменение стиля текста строки на основе атрибутов соответствующей записи.

Значения являются выражениями Python. Для каждой записи выражение оценивается с использованием атрибутов записи в качестве значений контекста, и если `` true``, соответствующий стиль применяется к строке. Другими значениями контекста являются `` uid`` (идентификатор текущего пользователя) и `` current_date`` (текущая дата в виде строки вида `` yyyy-MM-dd``).

{$name} can be one of the following bootstrap contextual color (danger, info, secondary, success or warning).

`` default_group_by``
имя поля для группировки задач по
`` consolidation``
имя поля для отображения значения консолидации в ячейке записи
`` consolidation_max``
словарь с полем «group by» в качестве ключа и максимальным значением консолидации, которое может быть достигнуто до отображения ячейки красным (например, `` {user_id: 100} ``)
`` consolidation_exclude``
имя поля, которое описывает, если задача должна быть исключена из консолидации, если задано значение true, в строке консолидации отображается полосатая зона
`` create``, `` edit``, `` plan``

allows disabling the corresponding action in the view by setting the corresponding attribute to false.

  • create: If enabled, a «+» button will be displayed while hovering on a time slot to create a new record in that slot, and if
  • edit: If enabled, the opened records will be in edit mode (thus editable)
  • plan: If enabled and edit enabled, a «magnifying glass» button will be displayed on time slots to plan unassigned records into that time slot.

Example

When you do not want to create records on the gantt view and the beginning and end dates are required on the model, the planning feature should be disabled because no record will ever be found.

`` Offset``
В зависимости от масштаба, количество единиц, добавляемых к сегодняшнему дню, для расчета периода по умолчанию. Примеры: смещение +1 на неделе default_scale откроет представление Ганта для следующей недели, а смещение -2 в месяце default_scale откроет представление Ганта 2 месяца назад.

имя поля, отображающего процент завершения события записи, от 0 до 100

`` precision``

JSON-объект, указывающий точность привязки для таблеток в каждом масштабе.

  • Возможные значения для шкалы `` day`` (по умолчанию: `` hour``):

    `` hour``: время записи соответствует полному часу (например, 7:12 становится 8:00)

    `` hour: half``: время записи достигает половины часа (например, 7:12 становится 7:30)

    `` час: четверть``: время записи достигает половины часа (например, 7:12 становится 7:15)

  • Возможные значения для шкалы `` week`` (по умолчанию: `` day: half``):

    day: records times snap to full days (ex: 7:28 AM becomes 11:59:59 PM of the previous day, 10:32 PM becomes 12:00 PM of the current day)

    `` day: half``: время записи достигает половины часа (например, 7:28 утра становится 12:00 вечера)

  • Возможные значения для масштаба `` month``: (по умолчанию: `` day: half``):

    day: records times snap to full days (ex: 7:28 AM becomes 11:59:59 PM of the previous day, 10:32 PM becomes 12:00 PM of the current day)

    `` day: half``: время записи достигает половины часа (например, 7:28 утра становится 12:00 вечера)

  • Шкала `` year`` всегда привязана к полному дню.

карта

Enterprise feature

Это представление может отображать записи на карте и маршруты между ними. Запись представлена булавками. Это также позволяет визуализировать поля из модели во всплывающем окне, привязанном к выводу записи.

Примечание

Модель, к которой применяется представление, должна содержать res.partner many2one, так как представление полагается на поля адреса res.partner и координаты для локализации записей.

Api

Представление использует api платформ данных о местоположении для извлечения фрагментов (фон карты), выполнения геообращения (преобразования адресов в набор координат) и извлечения маршрутов. Представление реализует два API, по умолчанию одно, openstreet map может извлекать tile и выполнять` geoforwarding`_. Этот API не требует токена. Как только действительный токен MapBox предоставляется в общих настройках, представление переключается на API Mapbox. Этот API быстрее и позволяет вычислять маршруты. Токен можно получить, зарегистрировавшись в MapBox

Структурные компоненты

Корневым элементом представления является `` <map> `` допускается несколько атрибутов

`` res_partner``
Содержит res.partner many2one. Если не указано, представление будет создавать пустую карту.
`` default_order``
If a field is provided the view will override the model’s default order. The field must be part of the model on which the view is applied not from res.partner
`` routing``
если `` true``, будут показаны маршруты между записями. Для представления все еще требуется действительный токен MapBox и как минимум две найденные записи. (т.е. в записях есть res.partner many2one, а у партнера есть адрес или действительные координаты)

Единственный элемент, разрешенный в пределах `` <map> `` элемент является `` <marker-popup> ``. Этот элемент может содержать несколько <field> Создавать элементы. Каждый из этих элементов будет интерпретирован как строка во всплывающем окне маркера. Атрибуты поля следующие:

`` Name``
Поле для отображения.
`` String``
Эта строка будет отображаться перед содержимым поля. Может использоваться как описание.

Никакие атрибуты или элементы не являются обязательными, но, как указано выше, если не указано res.partner many2one, представление не сможет найти записи.

Например вот карта:
<map res_partner="partner_id" default_order="date_begin" routing="true">
    <marker-popup>
        <field name="name" string="Task: "/>
    </marker-popup>
</map>

Pivot

Сводное представление используется для визуализации агрегатов в виде сводной таблицы. Его корневой элемент `` <pivot> `` который может принимать следующие атрибуты:

`` disable_linking``
Установите `` True``, чтобы удалить ссылки ячейки таблицы на представление списка.
`` display_quantity``
Установите значение `` true`` для отображения столбца Количество по умолчанию.
`` default_order``

Имя меры и порядок (asc или desc) для использования в качестве порядка по умолчанию в представлении.

<pivot default_order="foo asc">
   <field name="foo" type="measure"/>
</pivot>

Единственный разрешенный элемент в сводном представлении - это `` field``, который может иметь следующие атрибуты:

`` имя`` (обязательно)
имя поля для использования в представлении. Если используется для группировки (а не агрегирования)
`` String``
имя, которое будет использоваться для отображения поля в сводном представлении, переопределяет атрибут строки Python по умолчанию для поля.
`` type``

указывает, следует ли использовать поле в качестве критерия группировки или в качестве агрегированного значения в группе. Возможные значения:

`` row`` (по умолчанию)
групп по указанному полю, каждая группа получает свою строку.
`` col``
создает группы по столбцам
`` measure``
поле для агрегирования в группе
`` interval``
для полей даты и даты и времени группируются по указанному интервалу (`` день``, `` неделя``, `` месяц``, `` квартал`` или `` год``) вместо группировки по определенному дате / времени ( фиксированное второе разрешение) или дата (фиксированное дневное разрешение).
`` invisible``
если true, поле не будет отображаться ни в активных мерах, ни в выбираемых мерах (полезно для полей, которые не имеют смысла агрегированных, таких как поля в разных единицах, например, € и $).

Меры автоматически генерируются из полей модели; используются только агрегируемые поля. Эти меры также сортируются в алфавитном порядке по строке поля.

Предупреждение

Как и в графическом представлении, сводная таблица агрегирует данные о содержимом базы данных, что означает, что несохраненные функциональные поля нельзя использовать в сводных представлениях.

В сводном представлении поле `` field`` может иметь атрибут `` widget``, чтобы определять его формат. Виджет должен быть средством форматирования полей, из которых наиболее интересными являются `` date``, `` datetime``, `` float_time`` и `` financial``.

Например, сводное представление расписания может быть определено как:

<pivot string="Timesheet">
    <field name="employee_id" type="row"/>
    <field name="date" interval="month" type="col"/>
    <field name="unit_amount" type="measure" widget="float_time"/>
</pivot>

Qweb

Представления QWeb являются стандартными Qweb шаблоны внутри` arch` представления. У них нет определенного корневого элемента. Поскольку представления QWeb не имеют определенного корневого элемента, их тип должен быть указан явно (его нельзя вывести из корневого элемента поля `` arch``).

Представления QWeb имеют два варианта использования:

  • они могут использоваться в качестве шаблонов внешнего интерфейса, и в этом случае `` template`` следует использовать в качестве ярлыка.
  • они могут использоваться как фактические представления qweb (открытые внутри действия), и в этом случае они должны быть определены как обычные представления с явным `` типом`` (это не может быть выведено) и моделью.

Основные добавления qweb-as-view к базовому qweb-as-template:

  • У qweb-as-view есть особый случай для `` <nav> `` элемент с классом CSS `` o_qweb_cp_buttons``: его содержимое должно быть кнопками и будет извлечено и перемещено в область кнопок панели управления, `` <nav> `` сама будет удалена, это обходной путь для панели управления пока не существует

  • Рендеринг qweb-as-view добавляет несколько элементов в стандартный контекст рендеринга qweb:

    `` model``
    модель, с которой связано представление qweb
    `` Domain``
    домен, предоставляемый поиском
    `` context``
    контекст, предоставляемый поисковым представлением
    `` Records``
    ленивый прокси для `` model.search (domain) ``, это можно использовать, если вы просто хотите перебирать записи и не выполнять более сложные операции (например, группирование)

  • qweb-as-view также предоставляет дополнительные возможности рендеринга:

    • `` _qweb_prepare_context (view_id, domain) `` подготавливает контекст рендеринга, специфичный для qweb-as-view
    • `` qweb_render_view (view_id, domain) `` - это метод, вызываемый клиентом, который будет вызывать методы подготовки контекста и в конечном итоге `` env ['ir.qweb']. render () ``.

Поиск

Представления поиска отличаются от предыдущих типов представлений тем, что они не отображают * content *: хотя они применяются к конкретной модели, они используются для фильтрации содержимого другого представления (обычно агрегированных представлений, например reference/views/list`или :ref:`reference/views/list). Помимо этой разницы в случае использования, они определяются одинаково.

Корневым элементом поисковых запросов является `` <search> ``. Это не требует никаких атрибутов.

Возможные дочерние элементы представления поиска:

`` field``

поля определяют домены или контексты с предоставленными пользователем значениями. Когда создаются поисковые домены, доменные поля составляются друг с другом и с фильтрами с использованием ** И **.

Поля могут иметь следующие атрибуты:

`` Name``
имя поля для фильтрации
`` String``
метка поля
`` operator``

по умолчанию поля генерируют домены в форме: samp: [({name}, {operator}, {обеспеченное_значение})] где `` name`` - это имя поля, а `` said_value`` - значение, предоставленное пользователь, возможно, отфильтрованный или преобразованный (например, ожидается, что пользователь предоставит * метку * значения поля выбора, а не само значение).

Атрибут `` operator`` позволяет переопределить оператор по умолчанию, который зависит от типа поля (например, `` = `` для полей с плавающей точкой, но `` ilike`` для полей типа char).

`` Filter_domain``

полный домен для использования в качестве поискового домена поля, можно использовать переменную `` self`` для ввода предоставленного значения в пользовательский домен. Может использоваться для создания значительно более гибких доменов, чем один `` оператор`` (например, поиск по нескольким полям одновременно)

Если предоставляются оба оператора `` operator`` и `` filter_domain``, приоритет имеет `` filter_domain``.

`` context``

позволяет добавлять контекстные ключи, в том числе предоставленные пользователем значения (которые как для `` domain`` доступны как переменная `` self``, массив значений, например `` [id_1, id_2] `` для a: class: Поле ~ ~ odoo.fields.Many2one). По умолчанию поля не генерируют домены.

Примечание

домен и контекст являются включающими, и оба генерируются, если указан `` context``. Чтобы генерировать только значения контекста, установите `` filter_domain`` в пустой список: `` filter_domain = [] ``

`` groups``
сделать поле доступным только для определенных пользователей
`` widget``
использовать определенный виджет поиска для поля (единственный вариант использования в стандартном Odoo 8.0 - это виджет `` selection`` для полей Many2one)
`` Domain``
если поле может обеспечить автозаполнение (например, class: ~ odoo.fields.Many2one), фильтрует возможные результаты завершения.
`` Filter``

Фильтр - это предопределенный переключатель в окне поиска, его можно только включить или отключить. Его основными целями являются добавление данных в контекст поиска (контекст, передаваемый в представление данных для поиска / фильтрации) или добавление новых разделов в фильтр поиска.

Фильтры могут иметь следующие атрибуты:

`` строка`` (обязательно)
этикетка фильтра
`` домен`` (необязательно)
Odoo domain, будет добавлен к домену действия как часть поискового домена.
`` дата`` (необязательно)

имя поля типа `` date`` или `` datetime``. Использование этого атрибута позволяет создать набор фильтров, доступных в подменю меню фильтров. Предлагаемые фильтры зависят от времени, но не являются динамическими в том смысле, что их домены оцениваются во время создания панели управления.

Пример:

<filter name="filter_create_date" date="create_date" string="Creation Date"/>

Приведенный выше пример позволяет легко искать записи со значениями полей даты создания в один из следующих периодов (если текущий месяц август 2019 г.).

Create Date >
  August
  July
  June
  Q4
  Q3
  Q2
  Q1
--------------
  2019
  2018
  2017

Мути выбор вариантов разрешен.

`` default_period`` (необязательно)

имеет смысл только для фильтра с непустым атрибутом `` date``. определяет, какой период активирован, если фильтр входит в набор фильтров по умолчанию, активированный при инициализации представления. Если не указано, по умолчанию используется this_month.

Чтобы выбрать один из следующих параметров: сегодня, this_week, this_month, last_month, antepenultimate_month, четвертый_квартал, третий_квартал, второй_квартал, первый_квартал, этот_год, последний_год, antepenultimate_year.

Пример:

<filter name="filter_create_date" date="create_date" string="Creation Date" default_period="this_week"/>
`` context``

словарь Python, объединенный с доменом действия для генерации поискового домена

Ключ `` group_by`` может использоваться для определения группы, доступной в меню «Группировать по». Значение group_by может быть допустимым именем поля.

<filter name="groupby_category" string="Category" context = {'group_by': 'category_id'}/>

Определенная выше группировка позволяет группировать данные по категориям.

Когда поле имеет тип `` date`` или `` datetime``, фильтр генерирует подменю меню Group By, в котором доступны следующие параметры интервала: день, неделя, месяц, квартал, год.

Если фильтр входит в набор фильтров по умолчанию, активированный при инициализации представления, записи по умолчанию сгруппированы по месяцам. Это можно изменить с помощью синтаксиса date_field: interval, как в следующем примере.

Пример:

<filter name="groupby_create_date" string="Creation Date" context = {'group_by': 'create_date:week'}/>

Совет

Добавлено в версии 7.0.

Последовательности фильтров (без разделяющих их нефильтров) обрабатываются как включительно скомпонованные: они будут составлены с `` OR``, а не с обычным `` AND``, например

<filter domain="[('state', '=', 'draft')]"/>
<filter domain="[('state', '=', 'done')]"/>

если выбраны оба фильтра, будут выбраны записи, чье `` состояние`` равно `` черновик`` или `` сделано``, но

<filter domain="[('state', '=', 'draft')]"/>
<separator/>
<filter domain="[('delay', '<', 15)]"/>

если выбраны оба фильтра, будут выбраны записи, у которых `` state`` равно `` draft`` ** и ** `` delay`` меньше 15.

`` Separator``
может использоваться для разделения групп фильтров в простых видах поиска
`` group``
может использоваться для разделения групп фильтров, более читабельных, чем `` разделитель`` в сложных видах поиска
`` Searchpanel``

позволяет отображать панель поиска слева от любого просмотра нескольких записей. По умолчанию в представлениях списка и канбана включена панель поиска. Панель поиска можно активировать в других представлениях с атрибутом:

  • view_types a comma separated list of view types on which to enable the search panel default: „tree,kanban“

Этот инструмент позволяет быстро фильтровать данные на основе заданных полей. Поля указываются как прямые потомки `` searchpanel`` с именем тега `` field`` и следующими атрибутами:

  • `` name`` (обязательно) имя поля для фильтрации

  • `` select`` определяет поведение и отображение. Возможные значения

    • one (default) at most one value can be selected. Supported field types are many2one and selection.
    • multi several values can be selected (checkboxes). Supported field types are many2one, many2many and selection.
  • `` groups``: ограничения для определенных пользователей
  • `` строка``: определяет метку для отображения
  • `` icon``: указывает, какой значок используется
  • `` цвет``: определяет цвет иконки

Дополнительные необязательные атрибуты доступны в случае `` multi``:

  • `` домен``: определяет условия, которым должны удовлетворять записи comodel.

Домен может быть использован для выражения зависимости от другого поля (с select = one) панели поиска. Рассмотреть возможность

<searchpanel>
  <field name="department_id"/>
  <field name="manager_id" select="multi" domain="[('department_id', '=', department_id)]"/>
<searchpanel/>

В приведенном выше примере диапазон значений для manager_id (имена менеджера), доступных на экране, будет зависеть от значения, выбранного в настоящее время для поля `` Department_id``.

  • `` groupby``: имя поля комоделя (доступно только для полей many2one и many2many). Значения будут сгруппированы по этому полю.

  • `` disable_counters``: по умолчанию установлено значение false. Если установлено значение true, счетчики не будут вычисляться.

    Эта функция была реализована на случай, если производительность будет слишком плохой.

    Другой способ решения проблем с производительностью - правильно переопределить метод `` search_panel_select_multi_range``.

Поиск по умолчанию

Поля поиска и фильтры можно настроить с помощью `` context` действия, используя ключи: samp: search_default_ {name}. Для полей значение должно быть значением, которое нужно установить в поле, для фильтров это логическое значение или число. Например, предполагая, что `` foo`` является полем, а `` bar`` является фильтром, контекст действия:

{
  'search_default_foo': 'acro',
  'search_default_bar': 1
}

автоматически включит фильтр `` bar`` и будет искать в поле `` foo`` * acro *.

Числовое значение (от 1 до 99) может использоваться для описания порядка групп по умолчанию. Например, если `` foo`` и `` bar`` относятся к двум группам

{
  'search_default_foo': 2,
  'search_default_bar': 1
}

имеет эффект для активации сначала `` bar``, затем `` foo``.

  • Cras justo odio
  • Dapibus ac facilisis in
  • Morbi leo risus
  • Porta ac consectetur ac
  • Vestibulum at eros
