Edit on GitHub

Действия определяют поведение системы в ответ на действия пользователя: вход в систему, кнопка действия, выбор счета-фактуры, …

Действия могут быть сохранены в базе данных или возвращены непосредственно в виде словарей, например, в кнопочных методах. Все действия имеют два обязательных атрибута:

`` type``
категория текущего действия, определяет, какие поля можно использовать и как интерпретировать действие
`` Name``
краткое читаемое описание действия, может отображаться в интерфейсе клиента
`` binding_model_id``

если установлено, действие доступно в панели действий для данной модели

Примечание

Для действий с сервером используйте `` model_id``.

Клиент может получить действия в 4 формах:

  • `` False``
    если какой-либо диалог действий открыт, закройте его
  • Строка
    если :ref:`client action <reference/actions/client>`соответствует, интерпретировать как тег клиентского действия, иначе рассматривать как число
  • Число
    читать соответствующую запись действия из базы данных, может быть идентификатором базы данных или: term: external id
  • Словарь
    рассматривать как дескриптор действия клиента и выполнить

Наручники

Помимо двух обязательных атрибутов, все действия также имеют общие * необязательные * атрибуты, используемые для представления действия в контекстном меню произвольной модели:

`` binding_model_id``
указывает, с какой моделью связано действие
`` binding_type``

указывает тип привязки, который в основном зависит от контекстного меню, в котором будет отображаться действие

`` action`` (по умолчанию)
Указывает, что действие будет отображаться в контекстном меню Action связанной модели.
`` report``
Указывает, что действие появится в контекстном меню Print связанной модели.
`` binding_view_types``
разделенный запятыми список типов представлений, для которых действие появляется в контекстном меню, в основном «список» и / или «форма». По умолчанию `` список, форма`` (и список, и форма)

Действия с окном (`` ir.actions.act_window``)

Наиболее распространенный тип действия, используемый для представления визуализаций модели через views: действие окна определяет набор типов представлений (и, возможно, определенных представлений) для модели (и, возможно, конкретной записи модели).

Его поля:

`` res_model``
модель для представления взглядов для
`` Views``
список пар `` (view_id, view_type) . Второй элемент каждой пары - это категория представления (дерево, форма, граф, ...), а первый - необязательный идентификатор базы данных (или `` False). Если идентификатор не указан, клиент должен получить представление по умолчанию указанного типа для запрашиваемой модели (это автоматически выполняется с помощью fields_view_get()). Первый тип списка является типом представления по умолчанию и будет открыт по умолчанию при выполнении действия. Каждый тип представления должен присутствовать не более одного раза в списке
`` res_id`` (необязательно)
если представлением по умолчанию является `` form``, указывает загружаемую запись (в противном случае должна быть создана новая запись)
`` search_view_id`` (необязательно)
`` (id, name) `` pair, `` id`` - это идентификатор базы данных определенного вида поиска, который нужно загрузить для действия. По умолчанию выбирается стандартное представление поиска для модели.
`` target`` (необязательно)
должны ли представления быть открыты в основной области содержимого (`` current``), в полноэкранном режиме (`` fullscreen``) или в диалоговом / всплывающем окне (`` new``). Используйте `` main`` вместо `` current``, чтобы очистить крошки. По умолчанию `` current``.
`` context`` (необязательно)
дополнительные контекстные данные для передачи в представления
`` домен`` (необязательно)
фильтрация домена для неявного добавления ко всем просмотрам поисковых запросов
`` limit`` (необязательно)
количество записей для отображения в списках по умолчанию. По умолчанию 80 в веб-клиенте

Например, чтобы открыть клиентов (партнер с установленным флагом `` customer``) с представлениями списка и формы:

{
    "type": "ir.actions.act_window",
    "res_model": "res.partner",
    "views": [[False, "tree"], [False, "form"]],
    "domain": [["customer", "=", true]],
}

Или открыть представление формы определенного продукта (полученного отдельно) в новом диалоге:

{
    "type": "ir.actions.act_window",
    "res_model": "product.product",
    "views": [[False, "form"]],
    "res_id": a_product_id,
    "target": "new",
}

Действия в окне базы данных имеют несколько различных полей, которые должны игнорироваться клиентами, в основном для использования при составлении списка `` views``:

`` view_mode`` (по умолчанию = `` tree, form``)
Разделенный запятыми список типов представлений в виде строки (/! \ Без пробелов /! \). Все эти типы будут присутствовать в сгенерированном списке `` views`` (по крайней мере с `` False`` view_id)
`` View_ids``

M2M [# notquitem2m] _ для просмотра объектов, определяет начальное содержимое `` views``

Примечание

Представления Act_window также могут быть точно определены через `` ir.actions.act_window.view``.

Если вы планируете разрешить несколько представлений для вашей модели, предпочтите использовать ir.actions.act_window.view вместо действия `` view_ids``

<record model="ir.actions.act_window.view" id="test_action_tree">
   <field name="sequence" eval="1"/>
   <field name="view_mode">tree</field>
   <field name="view_id" ref="view_test_tree"/>
   <field name="act_window_id" ref="test_action"/>
</record>
`` View_id``
определенный вид добавляется в список `` views``, если его тип является частью списка `` view_mode`` и еще не заполнен одним из представлений в `` view_ids``

Они в основном используются при определении действий из Дата файлы:

<record model="ir.actions.act_window" id="test_action">
    <field name="name">A Test Action</field>
    <field name="res_model">some.model</field>
    <field name="view_mode">graph</field>
    <field name="view_id" ref="my_specific_view"/>
</record>

будет использовать представление my_specific_view, даже если это не представление по умолчанию для модели.

Состав на стороне сервера последовательности `` views`` выглядит следующим образом:

  • получить каждый `` (id, type) `` из `` view_ids`` (упорядочено по `` sequence``)
  • если `` view_id`` определен и его тип еще не заполнен, добавьте его `` (id, type) ``
  • для каждого незаполненного типа в `` view_mode`` добавьте `` (False, type) ``
[1] технически не M2M: добавляет поле последовательности и может состоять только из типа представления без идентификатора представления.

Действия URL (`` ir.actions.act_url``)

Разрешить открытие URL (веб-сайта / веб-страницы) с помощью действия Odoo. Можно настроить с помощью двух полей:

`` url``
адрес, который нужно открыть при активации действия
`` Target``
открывает адрес в новом окне / странице, если `` new``, заменяет текущий контент на страницу, если `` self``. По умолчанию `` new``
{
    "type": "ir.actions.act_url",
    "url": "https://odoo.com",
    "target": "self",
}

заменит текущий раздел контента домашней страницей Odoo.

Действия сервера (`` ir.actions.server``)

class odoo.addons.base.models.ir_actions.IrActionsServer(pool, cr)[исходный код]

Модель действий сервера. Действие сервера работает на базовой модели и предлагает различные типы действий, которые можно выполнить автоматически, например, с использованием правил базовых действий, вручную, добавив действие в контекстное меню «Дополнительно».

Начиная с Odoo 8.0, кнопка «Создать действие меню» доступна в виде формы действий. Создает запись в меню «Дополнительно» базовой модели. Это позволяет легко создавать действия сервера и запускать их в массовом режиме через интерфейс.

Доступные действия:

  • «Выполнить код Python»: блок кода Python, который будет выполнен
  • «Создать новую запись»: создать новую запись с новыми значениями
  • «Запись на запись»: обновить значения записи
  • «Выполнить несколько действий»: определить действие, которое запускает несколько других действий сервера

Разрешить запуск сложного серверного кода из любого допустимого места действия. Только два поля относятся к клиентам:

`` Id``
в базе данных идентификатор действия сервера для запуска
`` context`` (необязательно)
контекстные данные для использования при выполнении действия сервера

Записи в базе данных значительно богаче и могут выполнять ряд специфических или общих действий в зависимости от их «состояния». Некоторые поля (и соответствующие поведения) разделены между состояниями:

`` model_id``
Модель Odoo связана с действием.

`` State``

  • `` code``: Выполняет код Python, указанный в аргументе `` code``.
  • `` object_create``: Создает новую запись модели `` crud_model_id``, следуя спецификациям `` fields_lines``.
  • `` object_write``: обновляет текущую запись (записи) в соответствии со спецификациями `` fields_lines``
  • `` multi``: выполняет несколько действий, заданных аргументом `` child_ids``.

Государственные поля

В зависимости от его состояния, поведение определяется через различные поля. Соответствующее состояние указывается после каждого поля.

`` код`` (код)

Укажите часть кода Python для выполнения при вызове действия

<record model="ir.actions.server" id="print_instance">
    <field name="name">Res Partner Server Action</field>
    <field name="model_id" ref="model_res_partner"/>
    <field name="code">
        raise Warning(record.name)
    </field>
</record>

Примечание

Сегмент кода может определить переменную с именем `` action``, которая будет возвращена клиенту в качестве следующего выполняемого действия:

<record model="ir.actions.server" id="print_instance">
    <field name="name">Res Partner Server Action</field>
    <field name="model_id" ref="model_res_partner"/>
    <field name="code">
        if record.some_condition():
            action = {
                "type": "ir.actions.act_window",
                "view_mode": "form",
                "res_model": record._name,
                "res_id": record.id,
            }
    </field>
</record>

попросит клиента открыть форму для записи, если она удовлетворяет некоторому условию

`` crud_model_id`` (создать) (обязательно)
модель для создания новой записи
`` link_field_id`` (создать)
many2one to `` ir.model.fields``, указывает поле m2o текущей записи, для которого должна быть установлена вновь созданная запись (модели должны совпадать)
`` fields_lines`` (создание / запись)

поля для переопределения при создании или копировании записи. One2many с полями:

`` col1``
`` ir.model.fields`` для установки в соответствующей модели (`` crud_model_id`` для созданий, `` model_id`` для обновлений)
`` Значения ``
значение для поля, интерпретируемое через `` type``
`` тип`` (значение | ссылка | уравнение)
Если `` value``, поле `` value`` интерпретируется как буквальное значение (возможно преобразованное), если `` уравнение``, поле `` value`` интерпретируется как выражение Python и вычисляется
`` child_ids`` (multi)
Укажите несколько подэтапов (`` ir.actions.server``), которые необходимо выполнить в состоянии multi. Если подэтапы сами возвращают действия, последнее будет возвращено клиенту как собственное следующее действие мульти

Контекст оценки

Ряд ключей доступен в контексте оценки или окружающих действий сервера:

  • `` model`` объект модели, связанный с действием через `` model_id``
  • `` record`` / `` records`` запись / запись, на которой запущено действие, может быть аннулирована.
  • `` env`` Odoo Environment
  • `` datetime``, `` dateutil``, `` time``, `` timezone`` соответствующие модули Python
  • `` log: log (message, level = 'info') `` функция ведения журнала для записи отладочной информации в таблицу ir.logging
  • Конструктор `` Warning`` для исключения `` Warning``

Действия отчета (`` ir.actions.report``)

Запускает печать отчета

`` имя`` (обязательно)
полезно только как мнемоника / описание отчета при поиске в списке какого-либо типа
`` модель`` (обязательно)
модель вашего отчета
`` report_type`` (по умолчанию = qweb-pdf)
или `` qweb-pdf`` для отчетов в формате PDF или `` qweb-html`` для HTML
`` report_name`` (обязательно)
название вашего отчета (который будет именем вывода PDF)
`` groups_id``
Many2many field to the groups allowed to view/use the current report
`` multi``
если установлено значение `` True``, действие не будет отображаться в представлении формы.
`` Paperformat_id``
Many2one field to the paper format you wish to use for this report (if not specified, the company format will be used)
`` attachment_use``

если установлено значение `` True``, отчет генерируется только один раз при первом запросе, а затем повторно распечатывается из сохраненного отчета, а не создается заново каждый раз.

Может использоваться для отчетов, которые должны быть созданы только один раз (например, по юридическим причинам)

`` attachment``
выражение python, определяющее имя отчета; запись доступна как переменная `` object``

Действия клиента (`` ir.actions.client``)

Запускает действие, полностью реализованное в клиенте.

`` Tag``
клиентский идентификатор действия, произвольная строка, на которую клиент должен знать, как реагировать на
`` params`` (необязательно)
словарь дополнительных данных для отправки клиенту вместе с тегом действия клиента
`` target`` (необязательно)
должно ли действие клиента быть открыто в основной области содержимого (`` current``), в полноэкранном режиме (`` fullscreen``) или в диалоге / всплывающем окне (`` new``). Используйте `` main`` вместо `` current``, чтобы очистить крошки. По умолчанию `` current``.
{
    "type": "ir.actions.client",
    "tag": "pos.ui"
}

говорит клиенту запустить интерфейс Point of Sale, сервер не знает, как работает интерфейс POS.

Автоматизированные действия (`` ir.cron``)

Действия запускаются автоматически на заданной частоте.

`` Name``
Название автоматизированного действия (в основном используется при отображении журнала)
`` interval_number``
Количество * interval_type * uom между двумя исполнениями действия
`` interval_type``
Единица измерения частотного интервала («минуты», «часы», «дни», «недели», «месяцы»,
`` numbercall``
Количество раз, когда это действие должно быть выполнено. Если ожидается, что действие будет выполняться бесконечно, установите значение `` -1``.
`` Doall``
Boolean, уточняющий, нужно ли выполнять пропущенные действия в случае перезапуска сервера.
`` model_id``
Модель, по которой будет называться это действие
`` code``

Код содержания акции. Может быть простой вызов метода модели:

model.<method_name>()
`` nextcall``
Следующая запланированная дата выполнения этого действия (формат даты / времени)
  • Cras justo odio
  • Dapibus ac facilisis in
  • Morbi leo risus
  • Porta ac consectetur ac
  • Vestibulum at eros
